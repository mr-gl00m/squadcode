# v1.2: Tier 2 backlog

Status: queued for v1.2 or later. Build when there's a concrete use case, not before. Filtered against the Squad Code pitch.

These are the "valuable but not yet" items. None should be built speculatively; each has a clear "build when X happens" trigger.

## Plan mode (`EnterPlanMode` / `ExitPlanMode` tools)

The agent enters plan mode → most tools flip to `defer:true` so the model can think and outline without accidentally writing files. Exit plan mode re-enables them.

**Build when:** a user asks for "let me see what you're going to do before you do it" and the existing skill-based workaround stops being enough. The natural shape after the deferred tool catalog (Tier 1) lands.

**Estimated:** ~100 LOC once the deferred tool catalog exists. Most of the work is in the canonical request layer, not the tools themselves.

## Worktree mode (`EnterWorktree` / `ExitWorktree` tools)

Agent calls `EnterWorktree` → harness creates a fresh git worktree at `<repo>/.worktrees/<slug>`, swaps the agent's CWD into it, reloads any path-dependent system prompt sections. `ExitWorktree` swaps back, optionally discarding or merging the worktree.

**Build when:** the agent starts attempting risky multi-file refactors and you want a "try this and discard" affordance. Strong fit for harness use because you can safely let weaker local models experiment.

**Estimated:** ~250 LOC. Requires git worktree subprocess wrapping, slug validation, CWD swap discipline (every tool that depends on cwd needs to read it from context, not capture at construction time, this is already the case in Squad Code, so the existing tool layer is compatible).

**Watch for:** Windows worktree quirks. Test on the actual dev setup before landing.

## In-process subagent (one flavor only)

`AgentTool` spawns a sub-agent inside the same Node process, sharing the parent's `messagesRef` view via a child `QueryEngine`. The child gets a derived (more restrictive) `PermissionContext` and a constrained tool subset. Streams progress back to the parent.

**Build when:** the orchestrator needs to delegate a focused sub-task (e.g., "review this file and report back") without polluting the main conversation context. Distinct from the quad-conclave plan, that's *parallel* multi-model review; this is *sequential* delegation to one subagent.

**Skip:** the forked-process and remote-host flavors. Squad Code only needs in-process. Forked is a cache-hit optimization that matters at scale; remote requires server infrastructure that conflicts with the local-first charter.

**Estimated:** ~300 LOC. Hardest part is permission context derivation (subagent must not be able to grant itself more permissions than the parent).

## Read-only shell command classifier

Parse Bash/PowerShell pipelines and classify commands as read-only (`ls`, `cat`, `grep`, `find`, `git status`, `git log`, `npm ls`) vs mutating (`rm`, `mv`, `git checkout`, `npm install`). Auto-allow read-only; prompt for mutating.

**Build when:** users complain about permission-prompt fatigue, specifically that obvious read-only commands ("show me the git log") still require approval.

**Skip until then:** Squad Code's current `defaultPermission: "ask"` for Shell is the safe default. Misclassifying a mutating command as read-only is a real harm; misclassifying a read-only command as mutating is just an extra prompt.

**Estimated:** ~300 LOC including pipeline operator parsing (`|`, `&&`, `;`). The split-on-operators + classify-each-segment pattern is the canonical shape; lift it from any reasonable shell-parsing library or write it inline.

## Custom keybindings (single-key first, chord later)

User-configurable keybindings via `~/.squad/keybindings.json`. Single-key version: `{key: "ctrl+r", action: "history-search"}`. Chord version (later): `{chord: ["ctrl+k", "ctrl+s"], action: "save-session"}`.

**Build when:** a user asks for it. Squad Code uses Ink's `useInput` directly today, which is fine for a small fixed binding set. Adding configurability is for users who want to remap.

**Skip the chord version** until after single-key ships and gets used. Chords double the implementation complexity (the parser/resolver/template subsystems are ~800 LOC vs ~150 LOC for single-key).

## MCP entrypoint (Squad Code as MCP server)

Add `squad mcp` mode that runs Squad Code as an MCP server over stdio. Other agents (another Squad Code instance, or any MCP-aware client) can connect and call Squad Code's tools.

**Build when:** Squad Code starts being used inside other agent loops. Useful for compositional workflows (e.g., a high-level orchestrator delegates "find files matching X" to a Squad Code MCP server).

**Skip until then:** consuming MCP (calling external MCP servers as Squad Code tools) and exposing MCP (being callable as one) are different problems. Exposing is only useful once consuming exists, since the natural pattern is "two MCP-aware agents talk to each other."

**Estimated:** ~200 LOC using the official MCP SDK. The hard part is the permission boundary, an MCP client could be anything, so default-deny mutating tools and require explicit allowlist.

---

## Order suggestion

No strict order, pick whichever has a concrete use case first. If forced to rank by "most likely to come up first":

1. **Read-only shell classifier**, almost certain to be the first complaint after sustained use
2. **Plan mode**, natural follow-on once the deferred tool catalog is in
3. **In-process subagent**, needed before the quad-conclave or any "delegate this" workflow
4. **Worktree mode**, once the harness is stable enough to safely experiment in branches
5. **Custom keybindings**, only when asked
6. **MCP entrypoint**, only when there's a consumer
