# Rip-apart: codex

_OpenAI's official Codex CLI, Rust core, TS shim, native-binary distribution. Squad Code v1.1/v1.2 planning input._

## What it is

OpenAI's official local coding agent. Rust core in `codex-rs/` (120+ workspace crates), a TS shim in `codex-cli/` that downloads platform-specific native binaries from npm optional dependencies, an SDK in `sdk/typescript/` that shells back into `codex exec --json`, and a docs tree under `docs/` that is mostly redirects to `developers.openai.com/codex/*`. The agent talks to the OpenAI Responses API exclusively (the chat-completions wire was deliberately removed, see `codex-rs/model-provider-info/src/lib.rs:45`), authenticates against a ChatGPT account by default (PKCE device-code or browser-based) with API key as the fallback, and runs untrusted shell calls under a platform sandbox: Seatbelt on macOS, bubblewrap + seccomp on Linux (Landlock as legacy), a restricted-token + WFP scheme on Windows. AGENTS.md is the canonical agent-instructions file. The whole package is built with both Cargo and Bazel, plus a Nix flake. It is large, vendor-shaped, and the single best open-source reference for "agent shell safety done seriously."

## Architecture at a glance

**Workspace topology.** `codex-rs/Cargo.toml:1` lists ~120 workspace members. Signal-dense ones:

- `core/`, the agent loop, turn/session state, tool dispatch, MCP plumbing, compaction, hooks. Massive; the repo's own AGENTS.md begs contributors to stop adding to it (`AGENTS.md:64-75`).
- `protocol/`, wire-level `RolloutItem`, `SandboxPolicy`, `WritableRoot`, `PermissionProfile`, `HookEventName`, exec-output shapes, account/auth types. Shared across all crates.
- `sandboxing/`, cross-platform sandbox manager. Picks Seatbelt / LinuxSeccomp / WindowsRestrictedToken / None and produces a transformed argv (`codex-rs/sandboxing/src/manager.rs:48-62`, `seatbelt.rs`, `landlock.rs`).
- `linux-sandbox/`, standalone binary `codex-linux-sandbox` that the main binary self-invokes (via `arg0` dispatch) to apply `no_new_privs` + seccomp + bwrap (`codex-rs/linux-sandbox/src/landlock.rs:42-88`).
- `windows-sandbox-rs/`, Windows-specific restricted token + Windows Filtering Platform setup (`acl.rs`, `wfp.rs`, `deny_read_resolver.rs`, etc.).
- `execpolicy/`, Starlark-evaluated `.rules` policy DSL for command pre-filtering. Decisions are `Allow / Prompt / Forbidden` (`codex-rs/execpolicy/src/decision.rs:9-16`). Rules match either by prefix patterns (with allowed-alternatives tokens) or heuristics fallback.
- `apply-patch/`, the V4A patch format parser, applier, and a standalone `apply_patch` executable that the model can shell out to (`codex-rs/apply-patch/src/lib.rs:34-44`).
- `file-search/`, fuzzy filename search using `nucleo` (Helix's matcher) over a `ignore::WalkBuilder` traversal (`codex-rs/file-search/src/lib.rs:1-60`).
- `mcp-server/`, a JSON-RPC stdio server that exposes Codex itself as an MCP tool (so other agents can invoke Codex). `codex-mcp/` is the MCP client side.
- `login/`, PKCE-based device-code login flow plus a local `LoginServer` for the OAuth redirect (`codex-rs/login/src/lib.rs`).
- `responses-api-proxy/`, a privileged HTTP proxy that injects `Authorization: Bearer $OPENAI_API_KEY` and forwards only `POST /v1/responses`. Used for privilege separation so unprivileged users don't see the key (`codex-rs/responses-api-proxy/README.md`).
- `exec/`, the headless `codex exec` driver. Emits JSONL events to stdout (`thread.started`, `turn.started`, `turn.completed`, `item.started/updated/completed`, `error`), see `codex-rs/exec/src/exec_events.rs:9-37`.
- `tui/`, the interactive ratatui-based UI. Two-region streaming, table holdback, vim mode, configurable keymaps.
- `hooks/`, lifecycle hook dispatch. Nine event names: PreToolUse, PermissionRequest, PostToolUse, PreCompact, PostCompact, SessionStart, UserPromptSubmit, SubagentStart, Stop (`codex-rs/hooks/src/lib.rs:18-28`).
- `rollout/`, JSONL session persistence with a SQLite session index (`codex-rs/rollout/src/lib.rs`).
- `otel/`, OpenTelemetry integration. Defaults to a Statsig OTLP-HTTP exporter with a hardcoded API key (`codex-rs/otel/src/config.rs:9-36`).
- `code-mode/`, V8-backed JavaScript runtime that lets the model orchestrate tool calls by writing JS that calls `await tools.shell(...)` etc. (`codex-rs/code-mode/src/description.rs:12-34`).
- `cloud-tasks/`, interactive UI for Codex Cloud (`chatgpt.com/codex`) task management. Not local-relevant.
- `external-agent-sessions/` + `external-agent-migration/`, detects and imports Claude Code session history (`SOURCE_EXTERNAL_AGENT_NAME = "claude"` at `external-agent-migration/src/...:15`).

**Where the agent loop lives.** `codex-rs/core/src/session/` holds `session.rs`, `turn.rs`, `turn_context.rs`, `handlers.rs`, and the multi-agent dispatcher. `core/src/client.rs` is the `ModelClient` that holds per-session config (auth, provider, conversation id) and creates a `ModelClientSession` per turn. The Responses API is the only wire protocol, chat-completions is gone. A non-trivial detail: the Responses client speaks WebSocket as well as SSE, with a prewarm step that opens a WS connection before the first stream request (`core/src/client.rs:1-25`).

**How the sandbox is invoked.** On Linux, the main binary self-invokes via `arg0` dispatch with the basename `codex-linux-sandbox` (`sandboxing/src/landlock.rs:6`), passing a JSON-serialized `PermissionProfile` and a `--sandbox-policy-cwd` plus `--command-cwd`. The helper sets `PR_SET_NO_NEW_PRIVS`, installs a seccomp filter, then either uses bubblewrap (preferred) or Landlock (legacy). On macOS, the manager invokes `/usr/bin/sandbox-exec` directly (hardcoded path, not PATH-resolved, on the theory that an attacker who can swap `/usr/bin/sandbox-exec` already owns root, `sandboxing/src/seatbelt.rs:29`). On Windows, a separate `codex-windows-sandbox` binary handles restricted tokens, ACLs, and Windows Filtering Platform rules. The sandbox policy itself is a tagged enum (`protocol/src/protocol.rs:858-906`): `DangerFullAccess`, `ReadOnly { network_access }`, `ExternalSandbox { network_access }`, `WorkspaceWrite { writable_roots, network_access, exclude_tmpdir_env_var, exclude_slash_tmp }`. Writable roots carry their own `read_only_subpaths` and `protected_metadata_names` (`.codex`, `.git/hooks`, etc.) so a writable cwd doesn't accidentally let the agent rewrite git hooks (`protocol/src/protocol.rs:913-959`).

**How MCP is structured.** Two sides. `codex-rs/codex-mcp/` is the client, it maintains MCP server connections, manages tool name collisions, handles elicitation (the MCP "ask user a question" flow), and dispatches tool calls (`connection_manager.rs`). `codex-rs/mcp-server/` is the server, it exposes Codex as an MCP tool over stdio JSON-RPC so other agents can call Codex (`mcp-server/src/lib.rs`). Tool approval surfaces as MCP elicitations: `ExecApprovalElicitRequestParams`, `PatchApprovalElicitRequestParams`. The client uses `rmcp` (Rust MCP SDK).

**The Responses-API-first design.** `WireApi::Responses` is the only variant (`model-provider-info/src/lib.rs:50-65`). Built-in providers are OpenAI, Amazon Bedrock, Ollama, and LMStudio, and Ollama / LMStudio are configured with `WireApi::Responses` too (`lib.rs:420-434`), implying codex expects those servers to speak Responses-compatible wire. No Anthropic, no DeepSeek built-in. Users can add providers via `~/.codex/config.toml` but they have to speak Responses.

**Distribution.** `codex-cli/bin/codex.js` (the npm entry) maps `process.platform`+`arch` to a target triple, looks up an optional-dependency package like `@openai/codex-linux-x64`, and `spawn()`s the bundled native binary (`codex-cli/bin/codex.js:15-127`). Signal forwarding for SIGINT/SIGTERM/SIGHUP is explicit (`codex.js:202-215`). Homebrew, GitHub release tarballs, and DotSlash files are all supported install paths.

## Features worth stealing

**Cross-platform sandbox primitives.** This is the headline. `codex-rs/sandboxing/src/manager.rs` picks the right platform sandbox; the policy types in `codex-rs/protocol/src/protocol.rs:858-959` model writable-root-with-read-only-subpaths-and-protected-metadata-names cleanly. The Seatbelt base policy at `codex-rs/sandboxing/src/seatbelt_base_policy.sbpl:1-123` is a real-world hardened policy adapted from Chrome's renderer sandbox, deny-default, with carefully enumerated `sysctl-read` allowlists, pseudo-tty rules, IOKit and mach-lookup grants for the bits the model needs. The Linux side uses bubblewrap for filesystem isolation and seccompiler for the syscall filter; Landlock is kept as a legacy backup (`linux-sandbox/src/landlock.rs:42-88`). Windows is the heaviest lift: restricted tokens, ACLs, deny-read resolvers, and WFP rules (`windows-sandbox-rs/src/`).

**The V4A patch format.** `*** Begin Patch / *** Add File / *** Update File / *** Delete File / @@ context-header / *** End Patch` with `+`/`-`/` ` prefixed change lines (`codex-rs/apply-patch/apply_patch_tool_instructions.md:1-76`). Designed to be parseable when the model omits line numbers, with `@@ class Foo` style hunk headers for disambiguation. Tolerant of lenient mode for gpt-4.1 (`apply-patch/src/parser.rs:46-52`). Includes a streaming parser (`streaming_parser.rs`) so the TUI can show patch previews as the model emits tokens. The standalone executable lives at `codex-rs/apply-patch/src/standalone_executable.rs` and is invoked via `apply_patch <patch-body>`, the model writes a shell call to `apply_patch` rather than using a function-call tool, which avoids quoting hell in the function-call args.

**AGENTS.md as a convention.** Codex made this a brand. `docs/agents_md.md` redirects to `developers.openai.com/codex/guides/agents-md`. The internal hierarchical-message handling (search "child_agents_md" in `core/src/agents_md.rs`) supports nested AGENTS.md files with scope/precedence rules. Squad uses `CLAUDE.md` still; AGENTS.md adoption is a low-effort cross-tool compatibility win.

**Headless `codex exec` with JSONL event stream.** `codex-rs/exec/src/exec_events.rs:9-37` defines the stream-JSON event shape: `thread.started`, `turn.started/completed/failed`, `item.started/updated/completed`, `error`. Each item is a tagged enum (`AgentMessageItem`, `CommandExecutionItem`, `FileChangeItem`, `McpToolCallItem`, `WebSearchItem`, `TodoListItem`, `ReasoningItem`, `ErrorItem`). The TS SDK at `sdk/typescript/src/codex.ts:11-39` is literally a wrapper that spawns `codex exec --json` and parses the JSONL, the SDK is the CLI. That's a clean separation: one binary, two surfaces, machine-readable wire shared between them. Also: `codex exec --ephemeral` runs without persisting any session files, `--output-schema FILE` enforces a JSON Schema on the model's final response, `--output-last-message FILE` writes the final agent message to a file (`exec/src/cli.rs:31-86`).

**Two-region streaming with table holdback.** `codex-rs/tui/src/streaming/controller.rs:1-50` partitions the rendered markdown stream into a *stable region* (committed to scrollback) and a *tail region* (mutable, redrawn each tick). Tables are inherently non-incremental (adding a row reshapes column widths), so a pipe-table detector holds them back from stable until the stream finalizes. Resize handling re-renders at the new width and rebuilds the queued stable region. Squad's REPL streams deltas in-place but doesn't have this stable/tail discipline; lifting it would make wide-terminal resize during a stream sane.

**Permission profile JSON over a sandbox-helper boundary.** Instead of plumbing a struct through nested function calls, the main binary serializes `PermissionProfile` to JSON, passes it as `--permission-profile <json>` to the `codex-linux-sandbox` helper, which deserializes it on the other side (`sandboxing/src/landlock.rs:31-48`). Clean process boundary. This pattern fits Squad's future "external CLI subagent in `.squad/worktrees/<id>/`" plan, pass the agent's scope-lock as JSON across the worktree boundary.

**File search using `nucleo` + `ignore`.** `codex-rs/file-search/src/lib.rs:1-60` walks the tree with `ignore::WalkBuilder` (the same library ripgrep uses, so it honors `.gitignore` + `.ignore` + parent ignores) and matches with `nucleo` (Helix editor's matcher, very fast, supports indices for highlighting, `respect_gitignore` toggle, configurable thread count). Returns a snapshot with progressive results (`scanned_file_count`, `walk_complete`). Squad's Glob and Grep tools are JS-side `picomatch`/regex, a parallel fuzzy filename matcher would be a separate tool worth having for "find the file the user vaguely mentioned" scenarios.

**Device-code + PKCE login flow.** `codex-rs/login/src/device_code_auth.rs:1-100` and `codex-rs/login/src/server.rs` run a local HTTP server on a loopback port that catches the OAuth redirect, exchanges the auth code with PKCE verifier, and stores tokens on disk. The success page is embedded HTML (`login/src/assets/success.html`). This is the only sane way to do interactive auth without copy-pasting tokens; if Squad ever supports browser-based provider auth (Anthropic Console, OpenAI dashboard), this is the shape to copy. The token storage layer (`login/src/auth/storage.rs`) supports an `AuthCredentialsStoreMode` enum, file or keyring, so users can opt into OS keychain backing.

**Privilege-separated API key proxy.** `codex-rs/responses-api-proxy/` is a strict HTTP forwarder that reads `OPENAI_API_KEY` from stdin, allocates a stack buffer, copies the key in with a `"Bearer "` prefix, validates the format, moves it to a heap `String`, leaks it as `&'static str`, calls `mlock(2)` to keep it out of swap, and uses `HeaderValue::from_static().set_sensitive(true)` so the HTTP stack doesn't accidentally log it (`responses-api-proxy/README.md:96-110`). Only forwards `POST /v1/responses`, rejects everything else with 403. A privileged user starts the proxy; unprivileged users use it via a server-info JSON dropfile. Squad has no equivalent, it doesn't need one, but the pattern is good for any tool where one user controls the key and another runs the agent.

**Resume/fork by session id with a SQLite index.** Sessions are JSONL files under `~/.codex/sessions/` (`rollout/src/lib.rs:22-31`). A SQLite index (`rollout/src/session_index.rs`, `state_db.rs`) supports fast `find_thread_path_by_id_str`, `find_thread_meta_by_name_str`, name-based lookup. Cursor-based pagination is built into the `ThreadsPage` shape (`rollout/src/list.rs`). Squad already has SQLite-backed session listing, but codex's thread-name lookup (named conversations) is a useful idea Squad doesn't have. The `INTERACTIVE_SESSION_SOURCES` const (`rollout/src/lib.rs:24-31`) explicitly lists `Cli`, `VSCode`, `Custom("atlas")`, `Custom("chatgpt")`, codex tracks per-source session origin which is useful for filtering.

**External-agent session import.** `codex-rs/external-agent-sessions/src/detect.rs:19-50` walks an external agent's home (currently hardcoded `SOURCE_EXTERNAL_AGENT_NAME = "claude"` in `external-agent-migration/src/...:15`) for recent `.jsonl` session files, summarizes them, and offers to import. An import ledger (`ledger.rs`) records which sessions were already imported so the user isn't re-prompted. This is the "we want users to switch from Claude Code to Codex" feature, the import preserves user history. The pattern fits Squad too: detect `~/.claude/projects/*/*.jsonl` and offer to import, since Squad sessions are JSONL already.

**OAuth-style "request user input" as a tool call.** `codex-rs/core/src/tools/handlers/request_user_input.rs` and `request_user_input_spec.rs` expose a tool the model can call mid-turn to ask the user a structured question. The TUI catches the call as an elicitation and renders a prompt. Squad's permission flow is the only path the model has to interrupt; adding a structured "ask the user something" tool would let the model gather context without abusing the permission system.

**Hooks have nine events including PermissionRequest, PreCompact, PostCompact, SubagentStart, Stop.** `codex-rs/hooks/src/lib.rs:18-28`. Squad has six (PreToolUse, PostToolUse, PostToolUseFailure, SessionStart, SessionEnd, UserPromptSubmit). The interesting additions: `PermissionRequest` fires when permission is requested (useful for auto-approve scripting), `PreCompact`/`PostCompact` fire around compaction (useful for archiving the pre-compact transcript), `SubagentStart` fires per subagent spawn (useful for budget gating), `Stop` fires when the agent loop terminates regardless of cause (cleaner than `SessionEnd` for "did the run actually finish?").

**Output schema enforcement.** `codex exec --output-schema <FILE>` (`exec/src/cli.rs:53-54`) tells codex to constrain the final response to a JSON Schema. Squad's loop returns whatever the model emits; a `--output-schema` analog would help when Squad is invoked from automation that expects structured output.

**Skills as embedded + on-disk hybrid.** `codex-rs/skills/src/lib.rs:1-60` embeds a system skills directory (`include_dir!`) and installs it to `$CODEX_HOME/skills/.system` on first run, with a fingerprint marker file so reinstall is skipped on subsequent startups. User skills live in `$CODEX_HOME/skills/`. Squad loads skills from `~/.squad/skills/` and `./.squad/skills/` but doesn't ship any. Embedding a small set of system skills (project-kickoff, why-doc, etc., Cid already wrote them) would be a quality-of-life win for fresh installs.

**Configurable, context-aware keymaps.** `codex-rs/tui/src/keymap.rs:1-60` resolves keybindings via `context -> global fallback -> defaults` precedence, supports vim mode (`vim_normal`, `vim_operator`), and validates that no key is bound to two actions in the same context. Squad's REPL is keymap-fixed; this is overkill for v1.x but worth knowing about when Cid eventually wants vim mode.

**Apply-patch as a shell command.** Codex doesn't expose `apply_patch` as a function-call tool, it exposes it as a *shell command* that the model invokes via the `shell` tool: `shell {"command":["apply_patch","*** Begin Patch\n..."]}` (`apply_patch_tool_instructions.md:71-75`). This avoids the JSON-string-quoting hell of passing a multi-line patch as a function-call argument, because the patch body is `argv[1]` to a real executable. The `codex-apply-patch` binary self-resolves (looks for itself on PATH, falls back to a special `--codex-run-as-apply-patch` arg0 dispatch, `apply-patch/src/lib.rs:39-44`). Squad's ApplyPatch tool currently takes the patch body as a function-call string, which is fine in TS land but has caused parse trouble historically (cf BH-2026-05-10-001, BH-2026-05-10-003). Switching ApplyPatch to a shell-wrapped binary isn't realistic for a TS project, but the patch-as-shell-arg shape is interesting.

## What Squad Code already does better

**Real multi-provider.** Codex speaks one wire: Responses (`model-provider-info/src/lib.rs:50-65`). The chat-completions wire was deliberately removed in `discussions/7782`. Built-in providers are OpenAI, Bedrock, Ollama, LMStudio, and Ollama/LMStudio are configured assuming they expose Responses-API-compatible endpoints, which only works for some Ollama versions. Squad has four adapter kinds (`llm-chat`, `llm-message`, `llm-response`, `llm-local`) and supports DeepSeek, Anthropic Messages, OpenAI Chat + Responses, and arbitrary OpenAI-compatible local servers without forcing them through a wire-translation layer. Squad's user model catalog (`~/.squad/models.json`) lets users define arbitrary provider/model entries, codex's `model_providers` in config.toml is similar in shape but constrained to Responses wire.

**Local-first by default.** Codex's primary auth path is `Sign in with ChatGPT` (README.md:47-49). API key is the fallback. The "Codex Cloud" hooks (`codex-rs/cloud-tasks/`, `cloud-tasks-client/`, `chatgpt/` modules in `model-provider-info`) tie the CLI to OpenAI's hosted service. Squad has no hosted service, no account login, no cloud-task list. The provider is configured locally and that's it.

**JSONL audit chain with prev_hash.** Squad's audit DB (`~/.squad/audit.db`) chains rows with `prev_hash` for tamper-evidence (`SQUAD_CODE_FEATURES.md:160-164`). Codex's `rollout` crate writes JSONL but doesn't chain, it has a SQLite index for lookup but no cryptographic linkage. If you need to prove "this is exactly the sequence of decisions the agent made," Squad's design is stronger.

**Simpler distribution.** Squad is `npm install -g squad-code` and you're done. Codex is `npm install -g @openai/codex` + an optional-dependency native binary download per platform (`codex-cli/bin/codex.js:15-127`) + an alternative path via Homebrew + an alternative path via GitHub-release tarball + an alternative path via DotSlash. The native binary is fast but the install matrix is six paths instead of one.

**No telemetry by default.** Codex's `mcp-server/src/lib.rs:54` defines `DEFAULT_ANALYTICS_ENABLED: bool = true`. The OTel config defaults to a Statsig OTLP-HTTP endpoint at `https://ab.chatgpt.com/otlp/v1/metrics` with a hardcoded API key (`otel/src/config.rs:9-36`). Debug builds skip the export, but release builds opt-in by default. Codex also has a dedicated `analytics/` crate with detailed event tracking, `AcceptedLineFingerprint`, `CodexCompactionEvent`, `GuardianReviewedAction`, etc. (`analytics/src/lib.rs`). Squad has none of this. There is no `~/.squad/.../analytics-disabled` flag because there's no analytics.

**YOLO mode with checklist gate.** Squad's `--yolo` requires a `checklist.txt`/`CHECKLIST.md` in the cwd, appends it to the system prompt, rewrites deletes to archive moves, and rejects cd/Set-Location outside cwd (`SQUAD_CODE_FEATURES.md:127-141`). Codex's `--dangerously-bypass-approvals-and-sandbox` (`exec/src/cli.rs` near `dangerously_bypass_approvals_and_sandbox`) skips approvals and sandbox unconditionally, no checklist, no cwd rails, no archive-on-delete. The flag is appropriately scary but offers nothing between "ask every time" and "all bets off."

**Multi-provider permission system.** Squad's permission model handles arbitrary provider IDs and source-agent metadata cleanly because the catalog is data-driven. Codex's permission model assumes one provider, one auth flow, one set of expected tools, it has elaborate handling for `Allow / Prompt / Forbidden`, additional permission profiles, network policy, sandbox transformations, but the assumption underneath is OpenAI is the model and Codex is the runtime.

**No vendor lock-in.** This is the load-bearing one. Codex's UX is built around "Codex is your IDE agent that uses GPT-5 / o3 via the Responses API." Switch providers and most of the per-provider plumbing (websockets, prewarm, sticky routing tokens) becomes dead code. Squad's adapter shape treats provider switching as a first-class operation, `--provider deepseek` / `--provider ollama` / `--provider anthropic` change wire format and the rest of the loop keeps working.

## Anti-patterns / things to avoid

**The dual-language complexity.** Rust core + TS shim + Python SDK + TypeScript SDK + Bazel build (~120 BUILD.bazel files) + Cargo workspace + Nix flake + a `justfile` that orchestrates all of it. The repo's own AGENTS.md says "if you change `Cargo.toml`, run `just bazel-lock-update` to refresh `MODULE.bazel.lock`" (`AGENTS.md:33-36`). The maintenance tax of keeping Cargo and Bazel in sync is real. Squad's single Node + TS toolchain is one of its actual strengths. Do not introduce a Rust sidecar without a clear, irreducible reason (and "the sandbox" might eventually be that reason on Linux, but only if `bwrap` / `firejail` shell-out doesn't work).

**Native-binary distribution via optional-dependency packages.** `codex-cli/bin/codex.js:15-127` is the launcher. It maps `(platform, arch)` to a target triple, requires a per-platform npm package as an optional dependency, falls back to a local `vendor/` directory, throws if the right binary isn't found. Six platforms, six packages, six tarballs in every release. The performance is great; the install is brittle. Squad's "pure TS, runs anywhere Node runs" is a better default for an agent CLI.

**OpenAI account coupling.** The default login is `Sign in with ChatGPT` and `chatgpt/` is its own crate (`Cargo.toml:147`). The `model-provider-info::ModelProviderInfo` carries an `is_openai()` method and `supports_remote_compaction()` is true iff is_openai or Azure (`model-provider-info/src/lib.rs:385-395`). Behavior diverges based on "is this OpenAI." Don't replicate this, Squad's `llm-response` adapter shape covers Responses without making provider identity load-bearing.

**Opt-in-by-default telemetry.** `DEFAULT_ANALYTICS_ENABLED: true` (`mcp-server/src/lib.rs:54`), Statsig built into the default OTel exporter (`otel/src/config.rs:13-36`). Even with the debug-build skip and the env-var off-switch, the default is "phone home." Squad's local-first charter forbids this.

**Bazel + flake.nix + justfile.** `BUILD.bazel` at every crate, `MODULE.bazel`, `MODULE.bazel.lock`, `flake.nix`, `flake.lock`, plus the Cargo workspace. The repo even has a custom `defs.bzl` and `rbe.bzl` (remote build execution). This is OpenAI-internal-build-system shape leaking into open source. A new contributor has to install Rust + Bazel + Nix + Just before doing anything useful. Squad's `npm test` and `npm run build` should stay one-toolchain.

**A 120-crate workspace with a 500-LoC module limit per file.** The internal AGENTS.md says "target Rust modules under 500 LoC, excluding tests. If a file exceeds roughly 800 LoC, add new functionality in a new module instead" (`AGENTS.md:42-50`). The result is the workspace member list at `Cargo.toml:1-116`. Many of these crates are 200-500 LoC and exist mostly to break dependency cycles. The lesson is the opposite of what it looks like: micro-crating *makes* the dependency graph the load-bearing artifact, and changes ripple across `Cargo.lock` and `MODULE.bazel.lock` and a dozen `BUILD.bazel` files. Squad's current structure (one TS package, modules under `src/<area>/`) is a feature, not a phase to grow out of.

**The "core is bloated, please add elsewhere" plea.** `AGENTS.md:64-75` explicitly tells contributors not to add to `codex-core` because it has grown unmanageable. This is the same shape as the "do not inherit DeepSeek's monolith" finding from the prior batch, a codebase whose own contributors are asking each other to please stop adding to the central module is in mid-migration. Squad's `src/engine/`, `src/providers/`, `src/tools/`, `src/permissions/`, etc. boundaries should stay narrow on purpose so they never reach this state.

**ChatGPT-account auth.** `login/src/auth/` and the `chatgpt/` crate handle ChatGPT.com session tokens, account binding, plan detection (Plus/Pro/Business/Edu/Enterprise, README.md:49). This is product surface that exists because OpenAI wants Codex to be a vector for ChatGPT subscriptions. Don't replicate; even an API-key-based equivalent (OpenAI dashboard login that fetches an API key) is more product than tool.

**Wire-API removal as breaking change.** `model-provider-info/src/lib.rs:75` errors on `wire_api = "chat"` with a "no longer supported" message pointing to a GitHub discussion. Existing users had their `config.toml` break on upgrade. Codex can afford this because the upgrade story is `npm i -g @openai/codex@latest`. Squad's provider catalog should not break old configs on upgrade, additive evolution only.

**Hosted-task UI in the CLI.** `codex-rs/cloud-tasks/` is a ratatui app inside the CLI for managing tasks on `chatgpt.com/codex`. The CLI is now also a client for a hosted service. Don't add this kind of thing, `squad cloud-tasks` would be an entirely separate product and dilute the local-first promise.

## Concrete backlog inserts for Squad Code

1. **Adopt AGENTS.md as a recognized agent-instructions filename alongside CLAUDE.md.** Low effort. Squad already loads `CLAUDE.md`; adding `AGENTS.md` as an additional recognized name (with the same precedence rules) means Squad users in a mixed-tool team aren't forced to choose. Match codex's resolution order (project root → parents up to repo root). This is a cross-tool compatibility play, not a switch.

2. **Investigate `bwrap` / `firejail` shell-out for Linux shell-tool sandboxing.** Squad can't ship native Landlock from Node, but it *can* detect `bwrap` on PATH and wrap `Shell` invocations with a generated bwrap argv. Codex's `linux-sandbox` provides the structural template, pass a JSON permission profile across a process boundary, let the helper enforce it. The minimum useful version is: in YOLO mode on Linux, wrap shell commands with `bwrap --ro-bind / / --bind <cwd> <cwd> --proc /proc --dev /dev --unshare-all <cmd>` and refuse if `bwrap` isn't installed. macOS analog: `/usr/bin/sandbox-exec -f <profile.sbpl>`. Windows: nothing, recommend WSL2.

3. **Adopt the V4A patch format as an ApplyPatch fallback (or wire-compat).** Squad's ApplyPatch takes unified-diff bodies. The V4A format (`*** Begin Patch` / `*** Add File: <path>` / `*** Update File: <path>` / `@@ <ctx>` / `+/-/ ` / `*** End Patch`) was designed for model robustness, no line numbers, optional anchors, clear file-op headers. If Squad sees model-confusion symptoms in ApplyPatch (failed hunks, off-by-one errors on `+++ /dev/null`), adding V4A as an accepted alternate input is low-risk. Lift the streaming parser shape from `codex-rs/apply-patch/src/streaming_parser.rs` for live preview.

4. **Add `--output-schema <FILE>` to `squad --print`.** Codex's `exec --output-schema` (`exec/src/cli.rs:53-54`) enforces a JSON Schema on the model's final response. Useful for scripting (`squad -p "..." --output-schema spec.json | jq`). The implementation is a system-prompt addendum plus a final-response validator; provider-side enforcement (via response_format etc.) is a bonus where the catalog supports it.

5. **Add `--ephemeral` to skip session persistence.** Codex's `exec --ephemeral` (`exec/src/cli.rs:31-32`) is useful for CI runs, test fixtures, and one-shot probes where the JSONL transcript is noise. Squad currently always writes a session JSONL; adding an opt-out is one flag and a conditional in the session writer.

6. **Add `--ignore-user-config` / `--ignore-rules` opt-outs.** Codex's `exec --ignore-user-config` skips `$CODEX_HOME/config.toml` (`exec/src/cli.rs:34-36`). Useful for reproducible CI runs that should not pick up the user's defaults. Squad has `--provider` and `--model` flags but no way to say "ignore everything in `~/.squad/` for this run."

7. **Add a `PermissionRequest` hook event.** Codex's hook list (`hooks/src/lib.rs:18-28`) includes `PermissionRequest`, which fires whenever the agent asks for permission, useful for "auto-approve `npm test` if the cwd is `my-project/`" style local policy. Squad has `PreToolUse` but not a permission-specific event. Adding it is mostly a hook-name registration plus a fire-site in `src/permissions/`.

8. **Add `PreCompact` / `PostCompact` hooks.** Same source. Squad's `/compact` replaces history; a `PreCompact` hook could archive the about-to-be-summarized transcript to a sidecar (`~/.squad/sessions/<id>/pre-compact-<turn>.jsonl`) so users don't lose data on accidental compactions.

9. **Add a `request_user_input` tool.** Codex exposes this as a structured way for the model to ask the user a question mid-turn (`core/src/tools/handlers/request_user_input.rs`). Squad's only model-initiated interrupt is permission prompts; a structured "ask the user something" tool would let the model gather missing context cleanly. The implementation is small, a tool spec with a `prompt` field, a TUI handler that pauses and shows an input prompt, a return-value path that injects the user's response back into the turn.

10. **Embed system skills with an install-on-first-run fingerprint marker.** Codex's `skills/src/lib.rs:1-60` uses `include_dir!` to bake the system skills into the binary and writes them out to `$CODEX_HOME/skills/.system` only when the fingerprint changes. Squad already loads skills from `~/.squad/skills/`; ship a baseline set (project-kickoff, why-doc, ship-it, plus 2-3 Cid uses constantly) via `include_str!` or a static asset directory and install-on-first-run.

11. **Detect and offer to import Claude Code sessions on first run.** Codex's `external-agent-sessions/src/detect.rs` walks `~/.claude/projects/*/*.jsonl`, summarizes them, and offers import. Squad sessions are JSONL too, and the on-disk shape is similar enough that a translator is one schema-mapping function. Match codex's import-ledger pattern so re-running detection doesn't re-prompt for already-imported sessions.

12. **Lift the two-region streaming model for the REPL.** Codex's `tui/src/streaming/controller.rs` splits markdown into stable+tail with table holdback. Squad's Ink REPL streams deltas in place. The interesting parts to copy: (a) treat tables as non-incremental and only commit them at finalization; (b) re-render on resize without losing tail; (c) move committed lines to scrollback as a separate region. Worth doing during the v1.2 subagent UI work, when streaming-from-multiple-sources will already force a redesign.

13. **Add named threads (`squad sessions name <id> <name>`).** Codex's `rollout::find_thread_meta_by_name_str` (`rollout/src/session_index.rs`) supports name-based session lookup. Squad's `squad sessions` is id-based. Names make `squad --resume <name>` actually usable. Small SQLite migration plus a CLI command.

14. **Document a recommended `SUBAGENT.md` / `WORKTREE.md` convention before v1.2 ships.** When subagents land, every running subagent will need its scope-lock written somewhere the model can read. Codex's `AGENTS.md` precedent (nested files, child-context aware) is the right shape. Adopt it ahead of time rather than reinvent it later.

15. **Consider keyring-backed credential storage.** Codex's `keyring-store/` crate uses the OS keychain (macOS Keychain, Linux Secret Service, Windows Credential Manager) as an alternative to plaintext-on-disk for auth tokens. Squad's API keys come from env vars today; if Squad ever stores any provider credentials (Anthropic OAuth refresh tokens, OpenAI dashboard-issued temp keys), use the keyring as the default and plaintext only with explicit opt-in.

## Bottom line

Codex's standout contribution is sandbox seriousness, Seatbelt + bwrap + Landlock + Windows restricted-token + a writable-roots-with-protected-metadata-names policy model that has clearly been pressure-tested. The V4A patch format is the second-best idea, designed deliberately for model robustness and worth knowing about even if Squad doesn't adopt it wholesale. Everything else in codex worth stealing is small (AGENTS.md, `--ephemeral`, named threads, `request_user_input`, hooks for permission/compact); everything else worth knowing about is OpenAI-shaped (Responses-only wire, ChatGPT account login, opt-in-by-default telemetry, native-binary distribution, Bazel + Nix, the cloud-tasks UI inside the CLI). Squad should borrow the sandbox-helper-process pattern and the patch-format idea, ignore the vendor coupling, and stay multi-provider and local-first because that's the actual product differentiation.
