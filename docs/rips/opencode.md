# Rip-apart: opencode

_TS/Bun monorepo, Effect-based, daemon+client. Squad Code v1.2/v1.3 planning input._

## What it is

opencode (anomalyco/opencode) is the community-popular open-source AI coding agent, distributed as `opencode-ai` on npm with a Bun runtime and parallel installation paths for brew, scoop, AUR, Nix, and mise. It is structurally a daemon: a Bun-based HTTP/WebSocket server (`packages/opencode/src/server/server.ts:75`) that hosts the agent loop, the tool registry, the session store, and a permission bus, with several different clients attaching to it, a terminal UI built on OpenTUI/SolidJS that boots an in-process server by default (`packages/opencode/src/cli/cmd/run.ts:1-30`), a `bun opencode serve` headless mode (`packages/opencode/src/cli/cmd/serve.ts`), a SolidJS web app under `packages/app`, an Electron desktop wrapper under `packages/desktop`, a Zed integration via Agent Client Protocol (ACP, `packages/opencode/src/acp/README.md`), and a generated JS SDK at `packages/sdk/js` produced by running `@hey-api/openapi-ts` against the server's live OpenAPI spec (`packages/sdk/js/script/build.ts:14`). README is mirrored in 25 languages and the headline build target is `bun run --cwd packages/opencode`; there is no Go-based TUI client despite the popular reputation suggesting one, the "TUI" is OpenTUI rendering Solid components inside the same process as the agent runtime, and Go would only show up as a downstream embedder. AGENTS.md in the repo root (`AGENTS.md`) is the model-facing contract: terse house rules ("avoid try/catch where possible," "use Bun APIs," "do not extract single-use helpers preemptively"), and an SST/Cloudflare cloud story (`sst.config.ts`, `infra/app.ts`) handles share-session syncing through a Cloudflare Worker plus durable object. The agent is closer to a local daemon with optional cloud sync than a single-process CLI; the cloud pieces (sharable session URLs, account/auth, Cloudflare R2 bucket) are bolted onto a runtime that mostly works offline. Provider support runs through the AI SDK (Vercel `ai@6`) with first-class plugin hooks for auth/provider/model overrides.

## Architecture at a glance

The monorepo is a Bun workspace with 21 packages under `packages/` plus a single deferred SDK subtree under `sdks/vscode` (the JS SDK lives at `packages/sdk/js`). The relevant ones:

- `packages/opencode`, the runtime. ~80 source directories: agent, tool, session, provider, lsp, snapshot, share, skill, plugin, server, cli, mcp, permission, pty, sync, project, workspace, command, ide, audit, etc. ~1700 lines in `session/prompt.ts:1-1765`, ~500 lines in `tool/registry.ts:1-481`, ~460 lines in `agent/agent.ts:1-463`. The agent loop is Effect-driven (`effect@4.0.0-beta` everywhere) with Layer-based dependency wiring and `InstanceState` for per-workspace state. Tools are `Tool.Def` records assembled into a registry, filtered per model/provider/agent (`tool/registry.ts:322-367`), and dispatched through the AI SDK's `tool()` helper.
- `packages/sdk/js`, the generated client. The build script runs `bun dev generate > openapi.json` to dump the Effect HttpApi as OpenAPI, then `@hey-api/openapi-ts` generates a `fetch`-based client into `src/v2/gen/client/`. There's a post-codegen patch in `script/build.ts:50-58` that rewrites a buggy `ServerSentEventsResult` generic from the codegen tool, telling, because they're consuming SSE for streaming and the generator gets it slightly wrong. Two layered exports: `@opencode-ai/sdk` (v1) and `@opencode-ai/sdk/v2` (current, used by everything internal). The desktop, web, ACP server, and the in-process TUI all import the same `createOpencodeClient` factory and talk to either an in-process app handler (`Default = lazy(...)` in `server.ts:58-67`) or a real HTTP listener.
- `packages/plugin`, the plugin SDK. Type definitions for hooks: `event`, `config`, `tool`, `auth`, `provider`, `chat.message`, `chat.params`, `chat.headers`, `permission.ask`, `command.execute.before`, `tool.execute.before`, `tool.execute.after`, `shell.env`, `experimental.chat.messages.transform`, `experimental.chat.system.transform`, `experimental.session.compacting`, `experimental.compaction.autocontinue`, `experimental.text.complete`, `tool.definition` (`packages/plugin/src/index.ts:222-333`). Plugins are dynamic-imported npm packages, resolved through `packages/opencode/src/plugin/loader.ts:85-220`. Auth plugins can declare multi-step OAuth/API key prompts with conditional fields, opencode delegates Codex/Copilot/Cloudflare/Azure auth into named plugin loaders (`packages/opencode/src/plugin/codex.ts`, `github-copilot/`, etc.).
- `packages/app`, SolidJS + Vite web app (the `app.opencode.ai` host) sharing the SDK. `packages/desktop`, Electron wrapper around the same web app. `packages/ui` and `packages/storybook` are component libraries. `packages/web`, Astro docs site.
- `packages/core`, shared utilities (filesystem, hash, log, schema, glob, global paths, model/provider primitives). Imported as `@opencode-ai/core/*`.
- `packages/console`, Cloudflare worker that backs the hosted "console" (share sync, account dashboard).
- `packages/function`, Cloudflare Workers entrypoints. `packages/identity`, auth. `packages/extensions`, `packages/slack`, `packages/github`, `packages/containers`, `packages/enterprise`, adjacent services.
- `packages/llm`, provider-side AI SDK adapters (`@opencode-ai/llm`).
- `infra/`, SST entries for Cloudflare deployment: `app.ts` (Worker + R2 bucket + Astro + Static site), `console.ts`, `enterprise.ts`, `monitoring.ts`.

The agent loop sits in `packages/opencode/src/session/prompt.ts`. It's a single ~1700-line Effect function that builds messages, applies plugin transforms, selects tools from the registry per-agent (`tool/registry.ts:322`), calls `streamText` from the Vercel AI SDK with provider-specific options, handles SSE/canonical event mapping, persists parts to SQLite via Drizzle (`session/session.sql.ts`), tracks usage/cost/cache, and writes a snapshot before/after each edit turn so `session/revert.ts` can roll back to a known good state. The session-level state model is normalized into Drizzle tables (`SessionTable`, `PartTable`, `ProjectTable`, `SessionShareTable`). Storage is `bun:sqlite` per-process (`drizzle-orm/bun-sqlite`), with migrations under `packages/opencode/migration/`.

The client/server protocol is the OpenAPI HTTP surface generated from Effect's `HttpApi` definitions in `packages/opencode/src/server/routes/instance/httpapi/groups/*.ts`: session, control, event, experimental, file, global, instance, mcp, permission, project, provider, pty, question, sync, tui, workspace, v2. Sessions are addressed by ID; the server multiplexes by an `x-opencode-directory` header so a single daemon can host multiple project contexts. Streaming uses SSE; UI state and permission prompts use WebSockets (`websocket-tracker.ts`). The Go-or-not "TUI" question: there are no Go files in the repo (`find . -name '*.go'` is empty); the terminal UI is OpenTUI (`@opentui/core`, `@opentui/solid`) rendering Solid inside Bun, in-process or attached over HTTP.

The split's real shape: not "agent in TS / TUI in Go," but "one Effect-based runtime, many embedders (CLI, web, desktop, ACP, plugin host) sharing one HTTP+SDK protocol." That makes more sense for what they actually shipped, and it costs less than a true polyglot client.

## Features worth stealing

**Custom agents from markdown files with frontmatter** (`packages/opencode/src/config/agent.ts:106-130`). The single most portable steal. `.opencode/agent/*.md` or `agents/*.md` files (recursive) are scanned with Glob, parsed with `gray-matter`, validated through an Effect Schema, and inserted into the runtime agent registry. Frontmatter fields: `model`, `temperature`, `top_p`, `prompt`, `description`, `mode` (`subagent` | `primary` | `all`), `hidden`, `color`, `steps`, `permission` (full ConfigPermission schema), plus an open `options` bag for unknown keys (`agent.ts:77-96`). The deprecated `tools: { name: boolean }` map is auto-translated into the new `permission` shape with a `write/edit/patch → permission.edit` collapse. Built-in agents (`build`, `plan`, `general`, `explore`, `scout`, `compaction`, `title`, `summary`) are declared in code (`agent/agent.ts:127-278`) and user config can override per-key by name with merge semantics, `Permission.merge(defaults, fromConfig(value.permission), user)`. This is structurally identical to how Squad already loads skills and output-styles. Squad could share the markdown frontmatter parser and add `~/.squad/agents/*.md` + `./.squad/agents/*.md` with `mode`, `model`, `permissions`, and `prompt` keys without touching the agent loop core.

**Permission system as a ruleset, not a bool grid** (`packages/opencode/src/permission/` and `agent/agent.ts:103-122`). opencode's permission model is a ruleset of `(tool, pattern, action)` triples where action is `allow | ask | deny`, the pattern is glob-matched against tool arguments (file path for read/edit, command for bash, agent name for task), and rulesets are merged with `Permission.merge(...)` where later wins on conflict but specificity sorts within each layer. Their `defaults` set includes things like `read: { "*": "allow", "*.env": "ask", "*.env.*": "ask", "*.env.example": "allow" }` and `external_directory: { "*": "ask", [Truncate.GLOB]: "allow", [skillDirs]: "allow" }`, patterns are evaluated against absolute resolved paths. Squad already has specificity-sorted pattern-based permission rules, but the `external_directory` permission as a dedicated keyword (separate from `read`/`edit`) for cross-project access is a clean way to express "this tool wants to touch a file outside cwd; how should we treat it?" that Squad currently lumps into per-tool jail bypass.

**Subagent definition through the same agent loader** (`packages/opencode/src/agent/agent.ts:165-201`). The `general`, `explore`, and (experimental) `scout` subagents are defined identically to primary agents, same schema, same permission ruleset, and distinguished only by `mode: "subagent"`. The `task` tool reads the registry, filters for non-primary agents, and exposes them to the model as callable sub-tasks (`tool/registry.ts:307-320`). User-defined `.opencode/agent/foo.md` with `mode: subagent` immediately becomes a `@foo` invokable. Squad's planned subagent layer wants exactly this shape: subagents and primary agents share their loading path, and the calling tool surfaces them by `permission.evaluate("task", agentName, parent.permission)`. The native agents are tagged `native: true` and `hidden: true` for internal-only ones (`compaction`, `title`, `summary` are hidden primaries).

**Plan/Build agent split and the plan tool gate** (`agent/agent.ts:142-164`, `tool/plan.ts`, `tool/plan-enter.txt`, `tool/plan-exit.txt`). opencode ships two primary agents from the start: `build` (default, full edit access) and `plan` (read/grep/glob/bash with `edit: deny` everywhere except `.opencode/plans/*.md` and `~/.opencode/data/plans/*.md`). `plan_enter` / `plan_exit` are explicit tools the model invokes to transition; the plan agent has `plan_exit: "allow"` and the build agent has `plan_enter: "allow"`. Plan mode is gated by `OPENCODE_EXPERIMENTAL_PLAN_MODE=1` (`runtime-flags.ts:48`). This is exactly what Squad's checklist+YOLO mode partially does, but as an agent rather than a session flag. Cleaner.

**Snapshot-based revert outside the user's git** (`packages/opencode/src/snapshot/index.ts:1-762`). opencode tracks file state per turn into a side git repo under `~/.opencode/data/snapshots/...` with `core.longpaths=true`, `core.symlinks=true`, `core.autocrlf=false`. Each turn calls `snap.track()` to capture a tree, and `session/revert.ts:41-95` lets you revert to a prior turn by replaying patches into the user's worktree without touching the user's `.git`. The session schema stores `revert: { snapshot, ... }` so an unrevert can put state back. Exactly the pattern in the 2026-05-08 rip's DeepSeek section, but production-shipping. Squad's planned `revert_turn` should mirror this: per-project snapshot store under `~/.squad/snapshots/<project-hash>/`, git-as-storage (not because we love git, but because diff/patch primitives are already there), explicit revert and unrevert as session APIs.

**Bash arity tokenization for permission caching** (`packages/opencode/src/permission/arity.ts`, used by `tool/shell.ts`). When the user grants "always allow `npm test`" the system needs to match later `npm test` calls but not `npm publish`. opencode tokenizes the bash AST with tree-sitter (`tool/shell.ts:11`) and computes an arity prefix from leading positional args, then uses that as the permission match key. Squad already does "arity-prefixed command patterns" for shell grants; the structural improvement opencode has is doing it through tree-sitter rather than regex, which means PowerShell, bash, fish all parse correctly through the same code path (`web-tree-sitter` with bash and powershell grammars listed in `package.json` trusted deps). Worth migrating Squad to once `node-pty` and tree-sitter are already in the build (they aren't yet).

**Background job orchestration with task status visibility** (`packages/opencode/src/background/job.ts`, exposed through `tool/task_status.ts` + `task_status.txt`). The `task_status` tool is gated by `experimentalBackgroundSubagents` (`registry.ts:263`) and lets the model query status of in-flight subagent jobs. This is the runtime equivalent of "subagents need lifecycle events" from the previous batch, opencode has wired the model into the lifecycle so the parent can ask "is general agent done yet?" mid-loop rather than blocking. Worth eyeballing once Squad's subagent layer is past the `howl`/anguish event model.

**Plugin hook surface** (`packages/plugin/src/index.ts:222-333`). The full list: `event`, `config`, `tool`, `auth`, `provider`, `chat.message`, `chat.params`, `chat.headers`, `permission.ask`, `command.execute.before`, `tool.execute.before`, `tool.execute.after`, `shell.env`, `tool.definition`, plus experimental compaction/messages/system/text hooks. Plugins get the SDK client, the project info, a `$` Bun shell, the worktree path, and a workspace registration API for cloud workspace adapters. This is bigger than Squad needs but the hook names are well-chosen, `chat.params` for "modify temperature/topP/maxOutputTokens before send," `tool.definition` for "rewrite tool description/schema per-call," `shell.env` for "inject CWD-specific env vars" are all clean cut points. Squad currently has 6 hook events; the structurally interesting additions are `chat.params` (parameter mutation) and `tool.definition` (per-call schema mutation), which only make sense if users want to inject project-specific tool docs without forking the core tool.

**LSP integration as both background diagnostics and an LLM-callable tool** (`packages/opencode/src/lsp/lsp.ts:1-80`, `packages/opencode/src/tool/lsp.ts:11-113`). opencode launches LSP servers per language (`lsp/launch.ts`, `lsp/server.ts`, `lsp/language.ts`) and runs them as long-lived child processes. Diagnostics are collected via `Bus` events and folded into post-tool feedback. Separately, the LSP `tool/lsp.ts` exposes operations (`goToDefinition`, `findReferences`, `hover`, `documentSymbol`, `workspaceSymbol`, `goToImplementation`, `prepareCallHierarchy`, `incomingCalls`, `outgoingCalls`) to the model as a structured tool. Gated by `experimentalLspTool`. This is the DeepSeek post-edit-diagnostics idea, plus exposing LSP queries to the model itself for code understanding. The diagnostics injection part is more portable than the LLM-callable-LSP part; both are worth Squad's attention after v1.2.

**Per-agent permission rulesets for "external directory" reads** (`agent/agent.ts:93-102`, `tool/external-directory.ts`). The `readonlyExternalDirectory` ruleset whitelists the truncate output dir, `Global.Path.tmp/*`, and each loaded skill's directory tree. This lets a "read" tool access skill resources without requiring an explicit ask, while still asking for everything else outside cwd. Squad's filesystem jail goes all-or-nothing; the external_directory permission is a finer-grained "you can read outside the worktree if it's an approved location."

**OpenAPI-driven SDK generation** (`packages/sdk/js/script/build.ts`). The server's Effect HttpApi compiles to an OpenAPI spec at `bun dev generate`; `@hey-api/openapi-ts` emits a typed fetch client. The same machinery feeds the web app, the desktop wrapper, and any third-party embedder. The drawback is the build is fragile, they ship a manual patch (`build.ts:50-58`) to fix a codegen quirk in `ServerSentEventsResult`. But it's the cleanest local-first way to keep a network-addressable agent and let editors integrate without writing custom IPC. Squad already has a typed core; if the daemon path becomes real, generating a TS+Go+Rust client from a single OpenAPI doc is a smaller engineering lift than maintaining hand-written client libs.

**ACP (Agent Client Protocol) bridge** (`packages/opencode/src/acp/README.md`, `packages/opencode/src/acp/agent.ts`). opencode implements the Zed-driven Agent Client Protocol spec over JSON-RPC stdio (`opencode acp`). ACP `session/new` and `session/prompt` map onto opencode's existing session machinery; permission asks auto-approve (stub for now). This makes opencode a drop-in IDE agent for any ACP-supporting editor. ACP itself is community-defined (`agentclientprotocol.com`); supporting it costs ~one wrapper file. The model: keep the core session API; bolt protocol bridges (ACP for Zed, MCP server for Claude Desktop, custom JSON for whatever else) as thin adapters. Squad's MCP server roadmap entry could be one of several stdio-protocol adapters by the same pattern.

**SST/Cloudflare cloud story for share URLs only** (`infra/app.ts`, `packages/opencode/src/share/share-next.ts`). The cloud piece is genuinely narrow: a Cloudflare Worker (`Api`) plus an R2 bucket (`Bucket`) plus a static-site Astro deploy for the share viewer. Each session can be `share.create()`ed, which produces a URL containing a signed secret; the local daemon then queues writes (sessions, messages, parts, snapshot diffs, models) into the Cloudflare durable object via `SyncServer` (`infra/app.ts:36-48`). `OPENCODE_DISABLE_SHARE=1` kills the feature entirely. The interesting bit is the architecture: cloud is a write-only sync of session state, never a control channel. The local agent never receives instructions from the cloud. That's almost compatible with Squad's local-first thesis, but Squad still shouldn't ship it, see anti-patterns.

**Tool implementations to compare structurally**:
- `tool/edit.ts` (711 lines), find/replace, BOM preservation, line-ending preservation, mod-time recheck after permission preview, non-unique-match refusal. Squad already does all of this; opencode's approach is slightly more reactive (uses Effect's `useEffect` for the lock acquisition).
- `tool/apply_patch.ts`, unified-diff multi-file. Used preferentially for GPT models (`registry.ts:328-331`: `if (modelID.includes("gpt-") && !oss && !gpt-4) return usePatch`). Model-routed: GPT-5+ gets ApplyPatch, others get Edit/Write. Worth stealing the model-routing logic, Squad currently uses ApplyPatch and Edit interchangeably and lets the model decide.
- `tool/shell.ts`, tree-sitter parses, CWD-jail check, kill on timeout, `MAX_METADATA_LENGTH = 30_000` cap. Has `BashArity` for permission grant matching as noted above.
- `tool/grep.ts`, `tool/glob.ts`, ripgrep-backed (`@/file/ripgrep`). They bundle a `Ripgrep` service that locates an `rg` binary or downloads one. Squad uses Node-native regex; opencode is faster on large repos.
- `tool/skill.ts`, same shape as Squad's skills, scans `{skill,skills}/*.md`, loads through `Skill` service, exposes via the `skill` tool that returns `<skill_content name="...">` blocks the model can read.
- `tool/task.ts`, the subagent dispatcher. Resolves agent by name from the registry, evaluates `Permission.evaluate("task", agentName, parent.permission)`, refuses if denied. Returns subagent's final structured response to the parent.
- `tool/repo_clone.ts`, `tool/repo_overview.ts`, the experimental "scout" agent's tools for fetching dependency source. Cool, but `--allow-cloning-stuff-into-my-cache` is a feature surface Squad probably doesn't want.
- `tool/webfetch.ts`, `tool/websearch.ts`, `tool/mcp-websearch.ts`, webfetch is local fetch with timeout; websearch is provider-gated (Exa, Parallel, or the `opencode` first-party search via account).

## What Squad Code already does better

**Distribution.** `npm i -g opencode-ai` works but the runtime requires Bun. Their install path includes scoop, choco, AUR, paru, brew formula, mise, Nix, and a curl script as alternatives (`README.md:48-62`). Squad ships as a plain Node 22+ npm package with two binary names (`squad`, `squadcode`) and no second runtime. A user with `node --version` is done. opencode users get Bun and node-pty and tree-sitter native binaries pulled per-platform.

**Single-process simplicity.** opencode's run command boots an in-process server, mounts the OpenAPI handler, and the local TUI hits it through a `lazy()`-bound handler (`server/server.ts:58-67`). When it works that's fine; when it doesn't (server lifecycle bug, port conflict, mDNS publish failure, WebSocket tracker dispose order), the user sees an indirection layer their CLI didn't ask for. Squad's REPL is a single agent process; there's no extra hop, no protocol surface to debug. For the local-only single-user case this is the right tradeoff.

**Audit chain.** Squad has a SQLite audit DB with `prev_hash` tamper-evidence (`SQUAD_CODE_FEATURES.md:160-166`). opencode persists session/part/usage to Drizzle SQLite tables but does not chain hashes. If a user wants to verify "the session log was not tampered with after the fact" Squad answers, opencode doesn't.

**YOLO mode with checklist gate.** opencode has `plan` mode and per-agent permission rulesets, but it does not have Squad's "checklist file must exist in cwd before YOLO is armed" gate, the "rewrite `rm` into `mv -> .archive/<timestamp>/`" rewrite, and the "reject absolute paths outside cwd" sandbox (`SQUAD_CODE_FEATURES.md:126-141`). opencode's closest analog is `plan` agent which only denies; YOLO is a stronger affirmative permission with cheap rollback that opencode does not have.

**Provider-neutral catalog without an SDK dependency.** Squad's `default-models.json` plus four adapter kinds (`llm-chat`, `llm-message`, `llm-response`, `llm-local`) is dead-simple and doesn't require pulling Vercel's AI SDK. opencode delegates almost everything to `ai@6` (`session/prompt.ts:11`, `agent/agent.ts:4`), which means: when Vercel's AI SDK breaks (and it has, see `Bun.AI_SDK_LOG_WARNINGS = false` hack at `server.ts:18` and `prompt.ts:67`), opencode breaks. Squad owns its provider adapters.

**Local-first by default with no cloud opt-out flag needed.** opencode requires `OPENCODE_DISABLE_SHARE=1` to disable the cloud sync queue (`share-next.ts:19`). Squad does not have the queue at all. The cloud is not the default in opencode either, but the architecture assumes it could be. Squad doesn't have that ambient pressure.

**Smaller blast radius on a failed migration.** opencode's `data-migration.sql.ts` plus `JsonMigration` plus Drizzle's migration runner is real machinery; their state directory layout has changed across versions ("Remove versions older than 0.1.x before installing," `README.md:65`). Squad's migrations are linear SQLite migrations on session and audit tables. Squad isn't immune, but the surface is narrower.

**Output styles already implemented.** Squad already has the output-styles markdown-frontmatter loader that opencode doesn't ship as a first-class concept; opencode does the same thing through `agent/*.md` definitions, which conflates style with agent identity. Squad's split (skill = capability, output-style = formatting, agent = future) is cleaner.

## Anti-patterns / things to avoid

**Bun runtime as the implementation language.** opencode's source uses `Bun.file()`, `Bun.write()`, `bun:sqlite`, `$` shell, none of which run on Node without polyfills. They went all-in (`AGENTS.md:13` "Use Bun APIs when possible"). The cost is a forced runtime dependency for users, a divergent ecosystem (Bun's `node:` shims are not 100%), and a build target that has to be tested separately for every platform Bun supports. Squad is Node 22+ with no Bun. Keep it that way. If a Bun-specific perf win matters later, gate it behind a runtime check, not a global migration.

**SST/Cloudflare cloud-native deployment.** `sst.config.ts` plus 6 infra files plus a Stripe provider plus a Cloudflare R2 bucket plus a SyncServer durable object plus Honeycomb tracing, this is a meaningful operational footprint for what is sold as a coding agent. The cloud features exist because opencode the company wants a hosted SaaS offering. Squad's charter forbids hosted, telemetry, and remote sessions. Don't import any of this.

**Multi-language SDK generation overhead.** Even though only `sdks/vscode` exists today, the package.json catalog and `@hey-api/openapi-ts` setup imply an intention to generate Python/Go/Rust SDKs from the same OpenAPI. The cost: every breaking server change requires regenerating N clients, every codegen library bump requires patching the output (see `build.ts:50-58`), and at most one of those SDKs has real users. Squad does not have multiple clients to keep in sync; the in-process TS API is the only API. If Squad ever exposes an external surface, prefer a stable JSON-RPC over stdio (like ACP) over an OpenAPI HTTP layer with generated SDKs.

**Sharable cloud sessions.** `OPENCODE_DISABLE_SHARE=1` is the off switch; the default is on (`share-next.ts:19`). The session-share infrastructure leaks workspace contents, conversation, snapshots, and model usage to a Cloudflare durable object, user-controlled secret URLs, but still a remote copy of source. This is the single sharpest violation of Squad's local-first thesis. If sharing matters later, do it through a "squad export-session <id>" command that produces a portable JSONL file, not a hosted URL.

**The README-in-25-languages localization burden.** `README.md` plus `README.ar.md`, `README.bn.md`, `README.br.md`, `README.bs.md`, `README.da.md`, ... through `README.zht.md`. Twenty-five files to keep in sync with English on every update. Squad has one README. Keep it that way.

**The server/client split itself, mostly.** opencode's split exists for three reasons: hosted cloud sync (irrelevant for Squad), the web/desktop/Electron UIs sharing a network protocol (irrelevant for Squad, there is no web UI), and ACP/Zed integration (could be done as a separate adapter). Most of the operational complexity (port allocation, mDNS publish, WebSocket tracker, multi-tenant `x-opencode-directory` header dispatch, CORS middleware, lifecycle middleware) is not paying for itself in the local single-user case. If Squad ever needs daemon mode, a Unix socket with JSON-RPC framed messages is one order of magnitude simpler.

**Effect everywhere.** opencode is fully Effect: `Effect.gen`, `Layer`, `Context.Service`, `InstanceState`, `Schema`, `Schedule`, `Stream`. The DI graph is elegant once you read it but it's a steep learning curve for contributors, and the stack traces are Effect-shaped (`Effect.fn("Server.listen")`, attribute-decorated spans). Squad's plain TS with structured logging is more accessible. If Effect ever becomes attractive, adopt it surgically (one subsystem at a time) rather than in one rewrite.

**Drizzle ORM + bun-sqlite.** opencode uses Drizzle with `bun:sqlite` for everything (`storage/db.ts`, `session.sql.ts`, `share.sql.ts`, `project.sql.ts`). The ORM is fine; the runtime coupling to `bun-sqlite` is the issue. Squad uses `better-sqlite3` directly with hand-written queries and migrations. That's coarser but it doesn't make the database layer Bun-only.

**Heap snapshot toolkit as a dependency.** `heap-snapshot-toolkit` in the root package.json (`package.json:108`). This is the kind of operational tooling that gets pulled in once "the server keeps OOMing in production" becomes a real concern. Squad does not have a production. Don't proactively add ops tools for problems Squad has not had.

**Stripe, PlanetScale, Honeycomb providers.** `sst.config.ts:9-15` configures Stripe API key, PlanetScale, Honeycomb providers. Money, hosted DB, hosted observability. Three more cloud dependencies for a coding agent. Squad bills nothing, hosts nothing, traces nothing remote.

## Concrete backlog inserts for Squad Code

1. **Agent definitions from markdown frontmatter, mirroring skills.** `~/.squad/agents/*.md` and `./.squad/agents/*.md` with frontmatter keys `name`, `model`, `provider`, `description`, `mode` (start with `primary` only; `subagent` after v1.2), `permissions` (allow/deny/ask map), `temperature`, `steps`, `prompt` (body). Parser shared with skills + output-styles. Built-in default agent stays as-is; user agents override by name. The deprecated-tools translation (`tools: { write: true }` → `permission.edit = "allow"`) is worth copying verbatim from `config/agent.ts:77-96` because users will write the simpler form. Insert before v1.3.
2. **`plan` mode as a distinct agent rather than a CLI flag.** Define a built-in `plan` agent with `edit: deny`, `write: deny`, `apply_patch: deny`, `shell: ask`, `read/grep/glob/webfetch: allow`. Switch agents via `/agent plan` or `--agent plan`. Keep the existing `--print` mode independent. Reuse YOLO's checklist mechanism: plan mode's prompt addendum lists what plan mode forbids. Insert at v1.2.
3. **`external_directory` permission as a distinct permission kind.** Today Squad's filesystem jail is binary (in-cwd or rejected). Add an `external_directory` permission (allow/ask/deny + glob pattern) so users can grant tools access to `~/.squad/skills/*`, `~/.squad/output-styles/*`, `<truncated-output-dir>/*` without flipping the jail. Insert at v1.2 alongside agent loading.
4. **Side-snapshot revert under `~/.squad/snapshots/<project>/`.** Per-project git-as-storage (init bare repo, `core.longpaths=true`, `core.symlinks=true`, `core.autocrlf=false`); track-before-edit and track-after-edit in the agent loop; `/revert <turn-id>` slash command and `squad revert <session-id> <turn-id>` CLI; session schema field `revert: { snapshot, original_turn }` to support unrevert. Already in the post-2026-05-08 backlog; opencode's `snapshot/index.ts` is the reference impl. Insert after v1.2 (subagents need to interact with revert).
5. **Headless daemon mode v1.3+.** `squad serve --socket ~/.squad/squad.sock` exposing the session API over a Unix domain socket with framed JSON-RPC, not HTTP. Editors (Zed, VS Code, Helix) attach via the socket; the wire format is small (`session.new`, `session.prompt`, `session.events`, `permission.ack`, `tool.list`). Do not start with OpenAPI/HTTP. Once one editor lands, an OpenAPI surface might be worth adding to remove the editor's wrapper code; until then, JSON-RPC over stdio/sock is the cheapest stable contract. Treat ACP as a v1.4 thin wrapper around the same socket.
6. **LSP integration: diagnostics-after-edit injection first, LLM-callable LSP tool second.** opencode shows both work and the diagnostics-injection path is the higher-leverage one. After every successful `Edit`/`Write`/`ApplyPatch`, query the workspace LSPs for diagnostics on touched files, attach them to the next tool result as `<diagnostics>` blocks. The LLM-callable `lsp` tool (`goToDefinition`, `findReferences`, `documentSymbol`, etc.) is interesting but only matters once the model is being asked to refactor across files. Insert post-v1.2 as two separate features.
7. **Plugin hook surface expansion.** Squad currently has `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `SessionStart`, `SessionEnd`, `UserPromptSubmit`. Add `chat.params` (modify temperature/topP/maxOutputTokens before send) and `tool.definition` (modify tool description/JSON schema per-call). These are the two opencode hooks with concrete uses Squad's hooks can't express. `shell.env` (inject session-specific env vars into shell tool) is the third candidate. Insert at v1.3.
8. **Tree-sitter bash parsing for shell permission arity.** Squad's current shell-arity matcher is regex-based. opencode uses `web-tree-sitter` + bash/PowerShell grammars (`@/permission/arity`) which handles quoted args and PowerShell aliases without ad-hoc patches. Worth migrating only if Squad picks up another tree-sitter dependency for unrelated reasons.
9. **Model-aware tool routing for diff tools.** opencode dispatches `ApplyPatch` to GPT-5+ and `Edit/Write` to everything else (`registry.ts:328-331`). Squad currently exposes both to every model. Add a small model-to-tool-preference table in `default-models.json` so DeepSeek and Anthropic see Edit/Write, GPT-5+ sees ApplyPatch. Insert at v1.2 alongside catalog work.
10. **Background subagent task status.** Once subagents land, a `task_status` tool the parent can call mid-stream to check on in-flight subagents. opencode has this gated behind `experimentalBackgroundSubagents`; Squad's anguish/howl events already plan the lifecycle observability, `task_status` is the model-side accessor that pairs with it.

## Bottom line

opencode is the most architecturally adventurous of the agents in this rip batch, a Bun-based Effect-typed multi-client daemon with sharable cloud sessions, generated SDKs, plugin OAuth flows, durable objects, and an Electron desktop wrapper. Most of that adventure is the wrong shape for Squad's local-first single-user thesis, and the parts that are genuinely portable are the smallest parts of opencode: markdown-frontmatter agent definitions (`config/agent.ts:106-130`), the plan/build agent split, side-snapshot revert, and the LSP diagnostics-injection feedback loop. The architecturally adventurous parts that look transferrable (the daemon, the OpenAPI SDK generation, the cloud share) cost more operationally than they buy for a single local user, and Squad should defer them past v1.3 if it touches them at all. The salvageable kernel: agents-as-config-files plus side-snapshot revert plus diagnostics injection, each of which is an isolated, testable, weeks-not-months addition to the existing Squad runtime.
