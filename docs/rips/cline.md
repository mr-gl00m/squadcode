# Rip-apart: cline

_TypeScript, VS Code-first agent with a standalone gRPC runtime and a Commander-driven CLI. Squad Code v1.2/v1.3 planning input._

## What it is

Cline (formerly Claude Dev) is an Apache-2.0 coding agent that started as a VS Code extension and has grown a standalone gRPC runtime (`cline-core`), a JetBrains plugin, a separate `cline` npm CLI under `sdk/apps/cli/`, a `@cline/sdk` for plugin authors, and a Kanban web UI for parallel multi-agent runs. The shipped surface is enormous: ~40 providers in `src/core/api/providers/`, full MCP (stdio, SSE, streamable HTTP) with OAuth, Puppeteer browser tool, plan/act mode toggle, shadow-git checkpointing under VS Code globalStorage, subagent runner with its own context manager, focus-chain markdown todos, hooks (PreToolUse/PostToolUse/UserPromptSubmit and more), skills with frontmatter, ACP transport, scheduled agents via cron, Telegram/Slack/Discord connectors, plugin install from npm/git/path, and a sandbox mode via `--data-dir`. The noteworthy bits for Squad are the plan/act split, the granular auto-approve matrix, the shadow-git checkpoint store, the standalone gRPC runtime split, the subagent runner shape, and the evals/testing-platform layering. Everything else is either Squad's already-better local-first version or VS Code product surface that doesn't apply.

## Architecture at a glance

Top-level layout, `src/`, `webview-ui/`, `standalone/`, `sdk/`, `proto/`, `evals/`, `testing-platform/`, `tests/`. The agent loop lives in `src/core/task/index.ts` (3,764 lines, a god module called `Task`). Tool dispatch is in `src/core/task/ToolExecutor.ts:43`, which holds an `AutoApprove` instance and a `ToolExecutorCoordinator` and is constructed with ~30 callbacks back into `Task`. Per-tool handlers live in `src/core/task/tools/handlers/*Handler.ts`, one file per tool, each implementing `IToolHandler` / `IPartialBlockHandler` / `IFullyManagedTool`. Streaming state and abort flags live on `TaskState` in `src/core/task/TaskState.ts:6` (single mutable class, 30+ fields including `abort`, `didEditFile`, `currentlySummarizing`, `consecutiveMistakeCount`).

The webview is a Vite/React tree in `webview-ui/src/` and talks to the extension host via a custom gRPC-over-VS-Code-message-passing bridge. Proto files are in `proto/cline/*.proto` (`task.proto`, `mcp.proto`, `state.proto`, `checkpoints.proto`, `ui.proto`, `browser.proto`, `worktree.proto`, etc.). Generated code lands in `src/shared/proto/`, `src/generated/grpc-js/`, `src/generated/nice-grpc/`, `src/generated/hosts/` after `npm run protos`. Every UI-facing operation has a corresponding RPC and a `subscribeTo*` streaming endpoint (see `src/core/controller/checkpoints/subscribeToCheckpoints.ts`, `src/core/controller/ui/subscribeToPartialMessage.ts`). The whole architecture is "drive the UI by emitting proto events into the webview message bus."

MCP is wired through `src/services/mcp/McpHub.ts` (1,673 lines). It speaks all three MCP transports, `StdioClientTransport`, `SSEClientTransport`, `StreamableHTTPClientTransport` from `@modelcontextprotocol/sdk`, and watches `cline_mcp_settings.json` for live reload with chokidar. OAuth flows for remote MCPs are in `src/services/mcp/McpOAuthManager.ts`. Auto-approval for individual MCP tools is persisted on the server config record itself.

The standalone runtime lives in `src/standalone/cline-core.ts`. It boots a `host-bridge` (the VS Code "host" abstraction reimplemented over gRPC) on port `26041` and a `protobus-service` (the webview-to-extension gRPC) on port `26040`. Same `Controller` class as the VS Code path, see `src/standalone/protobus-service.ts:13`. A `SqliteLockManager` (`src/core/locks/SqliteLockManager.ts`) registers each running instance and cleans up orphaned folder locks, which is how multiple parallel cline-core processes coexist. The CLI in `sdk/apps/cli/` doesn't talk to `cline-core` directly, it uses `@cline/core` in-process. The standalone runtime is for JetBrains and other non-VS-Code hosts (and presumably the Kanban backend).

The `Task` class in `src/core/task/index.ts` is the entire agent loop: it holds the `ApiHandler`, `ToolExecutor`, `BrowserSession`, `McpHub`, `DiffViewProvider`, `ICheckpointManager`, `FocusChainManager`, `FileContextTracker`, `EnvironmentContextTracker`, `ModelContextTracker`, `ClineIgnoreController`, `CommandPermissionController`, `MessageStateHandler`, `ContextManager`, and `StreamResponseHandler`. Loop guards include consecutive identical tool detection (`src/core/task/loop-detection.ts:21`, soft=3 / hard=5) and a file-read dedup cache on `TaskState.fileReadCache`. Mode is read from `stateManager.getGlobalSettingsKey("mode")` per turn, `"plan"` or `"act"`, and toggled via the `togglePlanActModeProto` RPC in `src/core/controller/state/togglePlanActModeProto.ts`.

## Features worth stealing

**Plan/Act mode split.** `src/core/task/tools/handlers/PlanModeRespondHandler.ts:14` plus `ToolExecutor.PLAN_MODE_RESTRICTED_TOOLS` at `src/core/task/ToolExecutor.ts:291`. The mechanism:

- A single setting (`mode = "plan" | "act"`) on global state. Plan mode prepends `formatResponse.planModeInstructions()` to environment details (`src/core/task/index.ts:3756`).
- In strict plan mode, file mutating tools (`FILE_NEW`, `FILE_EDIT`, `NEW_RULE`, `APPLY_PATCH`) error out at the executor with a "switch to ACT MODE" message before they ever reach a handler.
- A `plan_mode_respond` tool is the only way the model talks back in plan mode. It pushes a `ClinePlanModeResponse` to the UI which renders as a chat bubble plus optional "options" buttons (`block.params.options` is parsed as a JSON array).
- The handler has an escape hatch: `needs_more_exploration="true"` lets the model continue planning without prompting the user. Without that hatch the model gets stuck mid-plan trying to call read tools after `plan_mode_respond` (single-tool-per-message rule).
- Toggling plan→act mid-response calls `config.callbacks.switchToActMode()` and returns `"[The user has switched to ACT MODE, so you may now proceed with the task.]"` as the tool result. The model continues the same conversation in act mode with the prior planning still in context.
- YOLO + plan mode auto-switches to act with no prompt (`PlanModeRespondHandler.ts:70`).
- Different system prompt variants per family, `src/core/prompts/system-prompt/variants/{generic,next-gen,native-next-gen,gpt-5,native-gpt-5,gemini-3,hermes,glm,xs}/config.ts`, each with its own plan/act prompt phrasing.

This is the single biggest borrow target. Squad has one mode and reaches for `TodoWrite` for everything that wants planning shape; explicit plan/act is a cleaner UX for "explore before you write."

**Granular auto-approve matrix.** `src/core/task/tools/autoApprove.ts:42`. Auto-approval is not one toggle, it's a record of action booleans on `autoApprovalSettings.actions`:

- `readFiles` / `readFilesExternally` (split by inside/outside workspace)
- `editFiles` / `editFilesExternally` (same split)
- `executeSafeCommands` / `executeAllCommands` (the "safe" tier auto-allows reads-only commands as classified elsewhere)
- `useBrowser`
- `useMcp`
- `maxRequests` (cap on auto-approved turns before re-asking)

`shouldAutoApproveToolWithPath` (`autoApprove.ts:122`) resolves the action path against the cwd and multi-root workspaces to decide local vs external. Squad's permission system has the right shape (allow/ask/deny with specificity-sorted pattern matching) but doesn't break out browser/MCP/safe-vs-all-commands into separate user-facing dials, and the "external" path discrimination is a useful idea, Squad's filesystem jail is binary (`SQUAD_PROJECT_PERMS` on/off), not in-vs-out-of-workspace with separate auto-approve tiers.

**Puppeteer browser tool.** `src/services/browser/BrowserSession.ts:38`. Single class wraps `chrome-launcher` and `puppeteer-core` with five primitives:

- `launchBrowser` / `launchLocalBrowser` (`:155`, `:212`), bundled Chromium fallback via `ensureChromiumExists()`
- `launchRemoteBrowser` (`:228`), connect via DevTools protocol to a remote Chrome on `DEBUG_PORT=9222`
- `navigateToUrl(url)` (`:481`)
- `click(coordinate)` (`:528`), coordinate is `"x,y"` literal
- `type(text)` (`:563`)
- `scrollDown` / `scrollUp` (`:571`, `:585`)
- Returns a `BrowserActionResult` with screenshot bytes (PNG, optionally WebP for models that support it) and console-log capture (`LoggerMessage[]`)

The tool handler is `src/core/task/tools/handlers/BrowserToolHandler.ts`. The system prompt teaches the model that browser actions are coordinate-based and that screenshots include a viewport ruler overlay. Worth weighing for Squad once a real use case appears, most "agent uses a browser" workflows actually want a headless fetch, which Squad doesn't have yet either.

**Shadow-git checkpoint system.** `src/integrations/checkpoints/CheckpointTracker.ts:50`. Cline maintains an isolated `.git` repo per workspace under `globalStorage/checkpoints/<cwdHash>/.git/` (`src/integrations/checkpoints/CheckpointUtils.ts:20`). Every tool execution that mutates files takes a snapshot commit (`saveCheckpoint` is called from `Task` and `ToolExecutor`). Restore semantics:

- `cwdHash` is computed from the absolute cwd path (so each project gets its own shadow repo)
- Nested git repos in the workspace are temporarily disabled during shadow-git operations (avoids submodule conflicts)
- Protected directories (`$HOME`, Desktop, Documents, Downloads) refuse checkpoint init, `validateWorkspacePath()` at `CheckpointUtils.ts:44`
- File locking via `CheckpointLockUtils.ts` (`tryAcquireCheckpointLockWithRetry`)
- Multi-root awareness in `MultiRootCheckpointManager.ts`
- Diff capability between any two checkpoint commits, restore-to-checkpoint replays files into the workspace
- Migrations live in `CheckpointMigration.ts` for repos that pre-date later schema changes

Squad's YOLO archive-on-delete is one slice of this. The full shadow-git model means a user can "revert to before turn 4" without touching their real git. The cost is that the shadow repo can balloon and the simple-git dependency footprint is real. The clean idea worth borrowing: **state lives outside the user's `.git` always**. Squad should follow that pattern when it adds `revert_turn`.

**MCP server management with auto-approve per-tool persistence.** `src/services/mcp/McpHub.ts`. Three transport types, OAuth via `McpOAuthManager.ts`, hot-reload of `cline_mcp_settings.json` via chokidar with a 300ms self-update flag (`McpHub.ts:62-76`) to avoid feedback loops on internal writes. Each server config has an `autoApprove: string[]` field listing tool names that skip the prompt for that server. The UI (`webview-ui/src/components/mcp/`) renders a server list with toggles, error states, restart buttons. Squad's roadmap notes MCP support is unshipped, when it lands, the file-watcher pattern and the per-tool auto-approve list are the right shape to copy. Avoid the OAuth complexity unless a real user asks.

**Task history with persisted diffs.** `src/core/task/index.ts` writes `api_conversation_history.json`, `context_history.json`, `ui_messages.json`, and per-task focus chain markdown (`focus_chain_taskid_<taskId>.md`) into a globalStorage task dir. `reconstructTaskHistory` (`src/core/commands/reconstructTaskHistory.ts`) replays a task. `exportTaskWithId` writes markdown export. Task IDs are ulids. The CLI variant adds session resumption via `cline --id <session-id>` and a separate session SQLite store under `~/.cline/`. Squad already has JSONL transcripts + SQLite index + `--resume`, same shape, cleaner implementation. The borrowable bit is the per-task focus chain file Cline writes alongside the transcript, which the user can hand-edit and Cline picks up on next turn via chokidar.

**Focus chain (live-editable todo list).** `src/core/task/focus-chain/index.ts:31`. A markdown checklist (`- [ ]` / `- [x]` lines) written to `focus_chain_taskid_<taskId>.md` in the task dir. The agent prompts itself to update it every N turns (`apiRequestsSinceLastTodoUpdate` on TaskState, `:67`). User edits to the file are picked up by a chokidar watcher and merged back into the conversation as a synthetic user message. Better than Squad's `TodoWrite` because it gives the user a writable artifact, not an opaque in-context list. The skill-discovered set is fed into the prompt at task start (`src/core/context/instructions/user-instructions/skills.ts`).

**Skills lockfile.** `skills-lock.json` at repo root. Tracks remotely-sourced skill packs with `source`, `sourceType: "github"`, `skillPath`, `computedHash` (SHA256 of the skill markdown). Skills are loaded from `.clinerules/skills/`, `.cline/skills/`, and the user's home equivalent. The lockfile lets the agent vendor in skill packs from GitHub and verify they haven't drifted. Squad's skill loader is local-first only, but the hash-pinning idea (you load a remote skill once, lock its hash, refuse to update silently) is a clean defense against supply-chain drift if Squad ever supports skill packs from URLs.

**Standalone gRPC runtime.** `src/standalone/cline-core.ts:25`. Worth understanding even if Squad won't copy it: the same `Controller` class that backs the VS Code extension is reused as a gRPC server (`protobus-service.ts:13`) with `addProtobusServices()` wiring all proto-defined services to the controller. A second gRPC server is the **host bridge** (`hostbridge-client.ts`), the file system, terminal, diff view, comment review, notifications are all proto-defined and the standalone runtime implements them via `ExternalCommentReviewController`, `ExternalDiffViewProvider`, `StandaloneTerminalManager`, etc. So any client that speaks the host-bridge protocol can host cline-core: JetBrains plugin, Kanban backend, eventually anything else. The unlock is **the host is pluggable**. Squad doesn't need this now (single CLI host), but if Squad ever grows a TUI vs CLI vs daemon-mode split, the lesson is: define the host surface as a protocol once, not as branched code paths.

**SqliteLockManager for instance coordination.** `src/core/locks/SqliteLockManager.ts`. `cline-core` registers each running instance in a SQLite `locks.db` with its protobus address and host address. Instances heartbeat via `touchInstance()`; orphaned folder locks (from crashed processes) get cleaned on startup (`cleanupOrphanedFolderLocks()`). Squad doesn't run multiple instances against the same workspace today, but if subagents ever share a worktree state, the heartbeat-and-orphan-clean pattern is the right primitive, not file-based pidlocks, not in-memory only.

**Subagent runner with isolated context.** `src/core/task/tools/subagent/SubagentRunner.ts:1-883`. Heavy file. Notable bits:

- A subagent has its own `ContextManager`, its own `ApiHandler`, its own usage tracking with cache-write/cache-read split, and its own `ToolExecutorCoordinator`
- `MAX_INITIAL_STREAM_ATTEMPTS = 3` with `INITIAL_STREAM_RETRY_BASE_DELAY_MS = 2000` exponential backoff on first-chunk failures
- `MAX_EMPTY_ASSISTANT_RETRIES = 3` if the model returns an empty assistant turn
- A separate `discoverAvailableSkills()` call at subagent start so the subagent gets its own skill set
- Progress updates streamed back via `SubagentProgressUpdate` (`SubagentRunner.ts:40`) with running/completed/failed states, latest tool call, and live stats, matches Cid's existing `howl`/lifecycle-event plan
- The `SubagentBuilder` (`src/core/task/tools/subagent/SubagentBuilder.ts`) reads from an `AgentConfigLoader.ts` that resolves agent definitions from rules files

The shape Squad's v1.2 subagent plan should match: each subagent is a fresh agent loop with its own context window, not a co-routine inside the parent's loop. Cline does this right. Cline's subagent does **not** use checkpoints (the parent owns the workspace), which is also the right call.

**Loop detection signature canonicalization.** `src/core/task/loop-detection.ts:37`, `toolCallSignature` strips ignored params (`task_progress` is filtered because it churns every turn) and uses `JSON.stringify(params, sortedKeys)` to canonicalize. Squad's identical-tool-call guard at 3 consecutive turns is close, but Squad doesn't filter metadata-only params from the signature. Cline's `IGNORED_PARAMS` whitelist is a tiny defensible borrow, adopt it for Squad's loop detector.

**Per-handler partial block streaming.** `IPartialBlockHandler` (`src/core/task/tools/ToolExecutorCoordinator.ts`). Each tool handler can implement `handlePartialBlock(block, uiHelpers)` to update the UI as the tool args stream in mid-call (the model is generating XML for the tool, the partial parse is rendered live). `PlanModeRespondHandler.handlePartialBlock` (`PlanModeRespondHandler.ts:24`) updates the chat bubble in real time. Squad's REPL is line-streamed; this finer-grained shape would only matter if Squad grows a TUI with per-message diff panes.

**Workspace-scoped command permissions.** `src/core/permissions/CommandPermissionController.ts:41`. Reads `CLINE_COMMAND_PERMISSIONS` env var with JSON allow/deny/allowRedirects. Parses commands via `shell-quote`, recursively validates subshells and segments split on `&&`, `||`, `|`, `;`. Detects newlines and unicode line separators outside quotes (`LINE_SEPARATOR_DESCRIPTIONS` at `:9`) as injection attempts. Squad's shell permission uses `arity-prefixed command patterns` per the feature inventory, Cline's stricter "deny redirects unless explicitly allowed, deny line-separator characters that could chain commands" is worth absorbing into Squad's `Shell` permission validation.

**Hook system.** `src/core/hooks/hook-executor.ts`. Cline's hooks live under `.clinerules/hooks/` as JSON files, run as subprocesses with stdin/stdout JSON IPC, support `PreToolUse`, `PostToolUse`, `UserPromptSubmit`, and a few others. A hook can return `cancel: true` to abort the tool, `contextModification: string` to inject a `<hook_context source="PreToolUse" type="...">` block into the next user message (`ToolExecutor.ts:416`). Squad already has the same six hook events. The borrowable bits: typed `contextModification` with `TYPE: content` prefix parsing (`ToolExecutor.ts:434`), and `setActiveHookExecution` / `clearActiveHookExecution` for clean cancellation tracking when the user aborts during a hook.

**Evals harness with pass@k metrics.** `evals/`. Three layers: contract tests (unit, `src/core/api/transform/__tests__/`), smoke tests (real LLM calls across providers via `evals/smoke-tests/`), E2E via `cline-bench` (twelve real bug-fix tasks as a git submodule). `evals/analysis/src/metrics.ts` computes pass@k and pass^k. Failure pattern classifier in `evals/analysis/src/classifier.ts` with patterns in `evals/analysis/patterns/cline-failures.yaml`. Baselines in `evals/baselines/` for regression detection. This is more eval rigor than gemini's offline-fixture approach, Cline runs the actual `cline` binary on `$PATH` against scenarios, not against fakes.

**Testing platform.** `testing-platform/`. A small package that drives `cline-core` via the gRPC adapters from outside. `testing-platform/adapters/grpcAdapter.ts`, `testing-platform/harness/runner.ts`. JSON spec files drive scenarios. Independent of the eval suite. The gRPC seam is what makes this possible; for Squad's local-first model, the equivalent is JSONL-driven scenarios that feed `squad --print` with golden tool fixtures (per gemini's lesson).

## What Squad Code already does better

**Provider-neutral catalog with adapter kinds.** Squad has four adapter kinds (`llm-chat`, `llm-message`, `llm-response`, `llm-local`) covering ~all OpenAI-compatible servers plus Anthropic and OpenAI Responses. Cline has ~40 distinct provider files in `src/core/api/providers/` (anthropic.ts, openai.ts, openai-codex.ts, openai-native.ts, deepseek.ts, gemini.ts, bedrock.ts, vertex via `gemini.ts`, ollama.ts, lmstudio.ts, openrouter.ts, vercel-ai-gateway.ts, cerebras.ts, groq.ts, mistral.ts, moonshot.ts, fireworks.ts, sambanova.ts, baseten.ts, hicap.ts, nebius.ts, qwen.ts, doubao.ts, claude-code.ts, qwen-code.ts, requesty.ts, together.ts, asksage.ts, aihubmix.ts, dify.ts, sapaicore.ts, oca.ts, minimax.ts, huawei-cloud-maas.ts, nousresearch.ts, litellm.ts, huggingface.ts, cline.ts). Each is its own class. The CLAUDE.md tribal-knowledge file documents that adding a new provider requires updates in seven places, three of which silently reset to Anthropic on miss. That's the cost of not having an adapter-kind abstraction. **Squad's three-or-four adapter kinds catch the same shapes with ~zero per-provider code**, and Squad's catalog override (`~/.squad/models.json`) is cleaner than Cline's seven-file gauntlet.

**Append-only JSONL audit chain with `prev_hash` tamper-evidence.** Squad's SQLite audit DB with hash-chained rows (per `src/sessions/audit-store.ts` referenced in feature inventory) is stronger than Cline's plain JSON file dumps in globalStorage. Cline writes `api_conversation_history.json` and `ui_messages.json` as plain JSON, atomically replaced via temp-file rename (`src/core/storage/disk.ts:30`). No chain hash, no migrations on the conversation files themselves. Squad's audit is forensic; Cline's is convenience.

**Atomic writes with BOM preservation and original line-ending preservation.** Cline does atomic writes via temp-and-rename in `src/core/storage/disk.ts:30`, but it doesn't preserve BOM or original CRLF/LF on edit (per Squad's feature list which calls out these properties on `Edit`). Cline's file-write path is the `DiffViewProvider` and `FileEditProvider`, VS Code aware, doesn't carry the BOM/EOL invariants into the standalone path cleanly.

**YOLO checklist gate.** Squad's YOLO requires a `checklist.txt` / `CHECKLIST.md` in cwd and appends it to the system prompt. Cline's yolo mode (`yoloModeToggled` setting) flips all auto-approve booleans to `true` and removes prompts; there's no required checklist artifact, no system-prompt rails. Cline's YOLO is "approve everything," Squad's YOLO is "approve everything **against this written checklist**, and archive deletions." Squad's is strictly safer for unattended runs.

**SSRF guard on local-provider URLs.** Squad's `llm-local` adapter refuses non-loopback Ollama URLs without `OLLAMA_ALLOW_REMOTE=1`. Cline's `ollama.ts` provider takes a URL and posts to it with no SSRF check. Cline relies on `getAxiosSettings()` from `src/shared/net.ts` for proxy support but doesn't gate the destination.

**Fewer moving parts.** No webview, no protobuf bridge, no Electron tax, no JetBrains plugin to keep in sync, no Kanban web UI, no scheduled-agent daemon, no Slack/Telegram connectors. Squad is one TS Node CLI. Cline is a 30+ subdirectory codebase with a `Controller` (`src/core/controller/index.ts`, 1,048 lines) that mediates ~120 proto RPCs across thirteen `.proto` files. Adding a single feature touches webview, controller, proto, generated bindings, state migrations, and tribal-knowledge doc updates (per the CLAUDE.md "Adding a New API Provider" section, this is a seven-file change that compiles but silently resets to Anthropic if you miss two).

**CLI-first.** Squad does not need VS Code, JetBrains, or a webview. Cline's CLI (`sdk/apps/cli/`) is real but its design center is still the IDE, and the CLI is increasingly the lightweight surface that calls into `@cline/core`. Squad starts at the CLI and stays there.

## Anti-patterns / things to avoid

**Do not adopt protobuf-over-IPC as the webview bridge.** It's the right answer when you need cross-language plugin clients (JetBrains, Kanban backend, third-party SDKs), but Squad has zero of those. The cost is real: every UI message round-trips through proto generation, every new chat-message variant requires updating `ClineSay` enum + `ExtensionMessage.ts` + `proto-conversions/cline-message.ts` + handler + webview rendering (per Cline's CLAUDE.md). Squad's "agent emits text and tool events into a JSONL transcript and a TUI reads them" is simpler and faster to evolve.

**Do not let `Task` become a god module.** `src/core/task/index.ts` is 3,764 lines. It mixes streaming state, abort handling, mode switching, focus chain, hooks dispatch, checkpoint management, environment details rendering, error retry policy, and tool result formatting. The class holds direct references to fifteen-plus collaborators. Squad's current shape, engine, providers, tools, permissions, sessions, audit, hooks, yolo as thin modules with cross-cutting types, should not collapse into a single `Task`. When subagents land, resist the urge to put them inside the same loop class.

**Do not adopt the "approve every MCP call" UX.** Cline's MCP auto-approve is per-tool persistent state, which trains users to click "always allow" on every server tool the first time it runs (because the alternative is to be prompted forever). The default Squad shape, pattern-based permission rules with allow/deny/ask actions and project-scoped persistence, gives the user a unified mental model. Don't bolt on a "MCP tool X is allowed for server Y" sub-system; pipe MCP tools through the same `permissions/` engine when MCP lands.

**Do not adopt VS Code coupling that leaks into "standalone."** Cline's `standalone/` directory shows the cost of retrofitting headless support. Files like `src/standalone/vscode-context.ts`, `vscode-context-utils.ts`, `hostbridge-client.ts` exist to mock out the VS Code API for the standalone runtime. The `ExternalDiffViewProvider`, `ExternalCommentReviewController`, `ExternalWebviewProvider` classes are no-ops or stubs for what VS Code does natively. Squad has no equivalent legacy to support, keep it that way.

**Do not adopt the gigantic webview-ui React tree.** `webview-ui/src/components/` is a full React codebase: chat row variants, MCP servers settings, browser session row, history list, settings panes for thirteen+ provider integrations, banner system, OAuth callback handling, onboarding wizard. Squad's Ink-based REPL with slash commands is 1/100th the surface and does the relevant work.

**Do not adopt the cron-scheduled-agent and Slack/Telegram/Discord/WhatsApp/Linear connector matrix.** Those are growth-channel features for a B2B product, not local-first agent features. They imply a always-on daemon (`cline hub`) and persistent OAuth tokens for messaging platforms. The connector surface in `sdk/apps/cli/src/connectors/` is real and active. Cid's local-first charter forbids most of this; skip it entirely.

**Do not adopt mass-telemetry as the feedback loop.** `src/services/telemetry/TelemetryService.ts` plus a PostHog provider and Sentry error reporting. `featureFlagsService` reads remote flags. Squad's local equivalent is `squad doctor`, JSONL transcripts, and local eval reports.

**Do not adopt the "tribal knowledge" tax.** Cline's `.clinerules/general.md` documents seven different multi-file ceremonies the contributor must perform to add a provider, slash command, system prompt section, state key, or RPC method without silent breakage. That tax is the proto layer's fault. Don't import it.

**Do not adopt the auto-approve "all tools" toggle.** `autoApproveAllToggled` and `yoloModeToggled` in `autoApprove.ts:66-86` are unconditional escape hatches that bypass everything. Cline's YOLO has no checklist rail; Squad's does. Resist any future PR that adds `--auto-approve-all` without keeping the checklist contract.

## Concrete backlog inserts for Squad Code

1. **Add a plan/act mode split.** Single setting (`mode = "plan" | "act"`) toggleable via `/plan` and `/act` slash commands and a `--mode plan|act` CLI flag. Plan mode prepends a "explore and ask, do not edit" instruction to the system prompt and rejects mutating tools (`Write`, `Edit`, `ApplyPatch`) at the executor with a "switch to act mode" message. Implement a `plan_mode_respond` tool with `options: string[]` for clickable choices. Auto-switching plan→act on user reply ("looks good, proceed") is a v1.2 polish item; the minimum viable shape is two distinct modes with a hard tool gate.
2. **Granular auto-approve dials in `.squad/settings.json`.** Replace the single permission decision with a record: `autoApprove: { readLocal, readExternal, editLocal, editExternal, shellSafe, shellAll, browser, mcp, web }`. "External" means outside the cwd jail. This unlocks a workflow where the user trusts the agent to read anywhere in the workspace but always asks before reading `~/.aws/`. Squad's pattern-based permission rules can implement this cleanly without a UI, emit it as a settings document with sensible defaults.
3. **Split shell permissions into `safe` and `all` tiers, with `safe` covering reads-only commands.** Borrow Cline's classification of read-only commands (`ls`, `git status`, `grep`, etc.) and let the user auto-approve the safe tier while keeping mutating commands gated.
4. **Add `IGNORED_PARAMS` to Squad's identical-tool-call guard.** Squad already aborts after three identical tool-call turns. Add a metadata-param filter so updating `task_progress` between otherwise-identical reads doesn't reset the counter. See `src/core/task/loop-detection.ts:27`.
5. **Tighten shell command validation.** Squad's shell permission patterns are arity-prefixed. Add the line-separator check (newlines, U+2028, U+2029, U+0085 outside quotes are command injection vectors) and the recursive subshell parser from `src/core/permissions/CommandPermissionController.ts:78` for commands that contain `(...)` or `$(...)`.
6. **Reject redirect operators by default in YOLO mode.** Cline's `allowRedirects: false` default in `CommandPermissionController` denies `>`, `>>`, `<`, `>&`, `<&`, `|&`, `<(`, `>(`. YOLO with redirects can clobber files outside the archive-rewrite path. Add this to Squad's YOLO shell sandbox.
7. **Move `revert_turn` to a side-snapshot directory** (already in the v1.2 plan per the precedent doc). Cline's shadow-git approach proves the pattern works at scale; Squad's version should be `~/.squad/snapshots/<project-hash>/.git/` with the same protected-directory refusal (`$HOME`, Desktop, Documents, Downloads).
8. **Add a focus-chain markdown artifact for multi-step tasks.** A `.squad/focus_<session-id>.md` file with `- [ ]` / `- [x]` checkboxes that the user can hand-edit and the agent picks up via chokidar on the next turn. This is the live-editable version of `TodoWrite`. The transcript still records the in-context list; the markdown file is the writable surface.
9. **Add hash-pinning for any future remote skill packs.** A `~/.squad/skills-lock.json` modeled on Cline's `skills-lock.json`, `{ name, source, sourceType, computedHash }`. Skip remote skills today; when they ship, do not skip the lockfile.
10. **Add a `--mode plan|act|yolo` flag.** Today Squad has `--yolo` standalone. Unifying mode into one selector with three discrete values (plus `zen` later if it makes sense) is cleaner than three orthogonal booleans.
11. **Add a typed `contextModification` from hooks.** Squad's hooks already fire and audit. Borrow Cline's `TYPE: content` prefix parsing from `ToolExecutor.ts:434` so hooks can return `WORKSPACE_RULES: ...` and have the agent see it as a typed `<hook_context source="PreToolUse" type="workspace_rules">` block in the next user message.
12. **Add `setActiveHookExecution` / `clearActiveHookExecution` for cancellable hooks.** When a hook subprocess is running and the user Ctrl-Cs, Squad should kill the subprocess and recover. Cline does this via `taskState.activeHookExecution: HookExecution | undefined` with an `AbortController`. Squad's hook runner is fire-and-forget today.
13. **Defer the standalone gRPC runtime indefinitely.** It pays off for multi-host (JetBrains, Kanban, web UI). Squad has no second host on the roadmap. The lesson is to **keep the agent loop pure of host concerns** so if a second host ever ships, the boundary is already clean, not to add gRPC now.
14. **Defer the Puppeteer browser tool.** Worth weighing only if a real Squad use case needs DOM-level interaction. For now, a `WebFetch` tool that does HTTP-only HTML retrieval is the right scope. Cline's browser tool is heavy: `chrome-launcher` bundles Chromium (~150MB), screenshot capture, WebP encoding, console log capture, viewport overlay. None of that is free.
15. **Add a `subagent` lifecycle event stream that matches Cline's `SubagentProgressUpdate` shape.** `{ status, latestToolCall, stats: { toolCalls, inputTokens, outputTokens, cacheRead, cacheWrite, totalCost, contextUsagePct } }`. This is the parent-visible side of `howl`/anguish; subagent stats flow to the parent's TUI without coupling the parent's agent loop to subagent internals.

## Bottom line

Cline's best contribution to Squad's roadmap is the **plan/act mode split** and the **granular auto-approve matrix**. Both are tiny in implementation cost and large in UX clarity, and neither requires copying Cline's webview/proto/host-bridge complexity. The shadow-git checkpoint shape validates Squad's existing plan to put `revert_turn` snapshots outside the user's `.git`. Everything else, the standalone gRPC runtime, the React webview tree, the forty-provider mat, the cron daemon and chat connectors, is product surface that fights Squad's local-first charter and should be left in cline-main untouched.
